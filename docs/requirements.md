# TradingAgents 多智能体股票分析系统 - 需求文档

> 生成时间：2025-12-25
> 版本：1.0.0
> 状态：草稿

---

## 项目概述

TradingAgents 是一个基于大语言模型的多智能体协作股票分析系统，集成到现有的 StockAnalysis 股票分析平台中。系统模拟真实投资机构的决策流程，通过四阶段流水线架构（分析师团队→研究员辩论→风险评估→总结输出）提供专业的股票分析服务。

系统支持多用户、多 AI 模型、MCP 工具的灵活配置，提供实时的分析过程可视化和流式报告输出。用户可以进行单股分析和批量分析，管理员可以配置系统级资源供所有用户使用。

### 核心特性
- 四阶段多智能体协作分析流程
- 支持多种 AI 模型接入（OpenAI 标准格式）
- 支持 MCP 协议工具动态接入
- 分析过程可视化
- 流式报告输出
- 多用户隔离与权限管理
- 公共模型资源池管理
- 任务队列与批量分析

## 需求规格说明

### 一、基础功能模块

#### 需求 1.1：AI 模型管理

**用户故事**：作为用户，我希望能够配置和管理 AI 模型连接，以便使用不同的 AI 服务进行股票分析。

**验收标准**：

1.1.1 当管理员添加系统级 AI 模型时，系统应验证 API 连接有效性并将模型配置存储到系统配置中。

1.1.2 当普通用户添加个人 AI 模型时，系统应验证 API 连接有效性并将模型配置存储到用户配置中。

1.1.3 当用户配置 AI 模型时，系统应要求提供：模型名称、提供商、API Base URL、API Key、模型 ID、并发限制、超时设置、温度参数。

1.1.4 当用户设置模型并发限制时，系统应确保该限制作为该模型的全局并发上限。

1.1.5 当用户测试 AI 模型连接时，系统应发送测试请求并在 5 秒内返回连接状态。

1.1.6 当用户启用或禁用 AI 模型时，系统应立即更新模型可用状态。

1.1.7 当普通用户访问 AI 模型管理页面时，系统应仅显示用户个人配置的模型列表。

1.1.8 当管理员访问 AI 模型管理页面时，系统应分别显示系统级模型和个人模型两个列表。

1.1.9 当用户删除 AI 模型时，系统应检查是否有进行中的任务使用该模型，若有则拒绝删除。

1.1.10 当用户配置模型温度参数时，默认值应为 0.5，可配置范围为 0.0 到 1.0。

---

#### 需求 1.2：MCP 服务器管理

**用户故事**：作为用户，我希望能够配置和管理 MCP 服务器，以便为智能体提供外部工具能力。

**验收标准**：

1.2.1 当用户添加 MCP 服务器时，系统应支持 stdio、SSE 和 HTTP 三种传输模式的配置。

1.2.2 当用户配置 stdio 模式 MCP 服务器时，系统应要求提供 command、args 和 env 配置项。

1.2.3 当用户配置 SSE 或 HTTP 模式 MCP 服务器时，系统应要求提供 URL 和认证信息。

1.2.4 当用户保存 MCP 配置时，系统应验证 JSON 格式正确性并提供格式化功能。

1.2.5 当用户测试 MCP 服务器连接时，系统应尝试连接并获取工具列表，在 10 秒内返回连接状态和可用工具数量。

1.2.6 当 MCP 服务器连接失败时，系统应自动标记该服务器为不可用状态。

1.2.7 当用户启用或禁用 MCP 服务器时，系统应立即更新服务器可用状态。

1.2.8 当用户配置工具自动批准列表时，系统应存储 autoApprove 工具名称列表。

1.2.9 当管理员添加系统级 MCP 服务器时，系统应使该服务器对所有用户可见但不可修改。

1.2.10 当普通用户访问 MCP 管理页面时，系统应显示系统级 MCP（只读）和个人 MCP（可编辑）两个列表。

---

#### 需求 1.3：智能体配置管理

**用户故事**：作为用户，我希望能够自定义智能体配置，以便根据需求调整分析策略和提示词。

**验收标准**：

1.3.1 当用户访问智能体配置页面时，系统应显示四个阶段的智能体配置选项卡。

1.3.2 当用户配置第一阶段智能体时，系统应允许添加、删除、修改分析师智能体。

1.3.3 当用户添加第一阶段智能体时，系统应要求提供：标识符（slug）、名称、角色定义（提示词）、使用场景说明、工具权限组。

1.3.4 当用户配置智能体工具权限时，系统应允许选择启用的 MCP 服务器和本地工具。

1.3.5 当用户配置第二、三、四阶段智能体时，系统应仅允许修改提示词，不允许添加或删除角色。

1.3.6 当用户修改智能体提示词时，系统应提供变量占位符说明（如 {stock_code}、{trade_date}）。

1.3.7 当新用户首次访问智能体配置时，系统应从系统默认模板初始化用户配置。

1.3.8 当用户重置智能体配置时，系统应恢复到系统默认模板。

1.3.9 当用户保存智能体配置时，系统应验证配置完整性并存储到用户配置数据库。

1.3.10 当用户为智能体选择 AI 模型时，系统应允许为不同阶段指定不同的 AI 模型。

---

#### 需求 1.4：第一阶段智能体架构

**用户故事**：作为系统，我希望第一阶段智能体采用工厂模式创建，以便支持动态配置和灵活扩展。

**验收标准**：

1.4.1 当系统创建第一阶段智能体时，应使用智能体工厂类根据配置动态创建。

1.4.2 当工厂创建智能体时，应使用通用模板类，仅替换提示词内容。

1.4.3 当第一阶段所有智能体执行完成时，工厂应将所有报告汇总后传递给调度系统。

1.4.4 当第一阶段执行过程中，工厂应暂存每个智能体的报告，直到所有智能体完成。

1.4.5 当用户添加新的第一阶段智能体时，仅需修改配置文件，无需修改代码。

---

### 二、核心功能模块

#### 需求 2.1：单股分析功能

**用户故事**：作为用户，我希望能够对单个股票进行深度分析，以便获取详细的投资建议。

**验收标准**：

2.1.1 当用户提交单股分析请求时，系统应创建分析任务并立即开始执行。

2.1.2 当分析任务开始执行时，系统应在用户打开分析详情页面时建立 WebSocket 连接推送实时状态。

2.1.3 当第一阶段智能体执行时，系统应根据并发配置并行执行多个分析师。

2.1.4 **[实时追踪]** 当智能体调用工具时，系统应通过 WebSocket 实时推送工具名称、输入参数、执行状态（运行中/完成）和输出结果，以便用户了解 AI 的操作细节。

2.1.5 **[增量更新]** 当**任意一个**分析师生成报告时，系统应立即存储该报告并通过 WebSocket 推送 `REPORT_READY` 事件，前端应立即显示该报告，无需等待所有分析师完成。

2.1.6 当第一阶段所有智能体完成时，系统应自动进入下一启用阶段。

2.1.7 **[任务终止]** 用户可以在任务执行的任何阶段点击“停止分析”按钮。系统收到请求后，应立即停止后续智能体的启动，保存当前已完成的报告，并将任务状态更新为“已停止”。

2.1.8 当第四阶段总结智能体执行时，系统应通过 SSE 流式输出最终报告。

2.1.9 当用户中断分析任务时，系统应停止后续智能体执行并保留已完成的报告。

2.1.10 当分析任务异常终止时，系统应记录错误信息并标记任务状态为失败。

2.1.11 当分析任务完成时，系统应生成包含推荐结果（买入/卖出/持有）、买入价格、卖出价格的完整分析结果。

2.1.12 当用户查看分析过程时，系统应显示当前执行阶段、智能体状态和工具调用历史。

2.1.13 **[历史查看]** 无论是“进行中”还是“已完成”的任务，用户都应能进入详情页查看。对于进行中的任务，显示实时进度；对于已完成/已停止的任务，显示最终状态和所有已生成的报告。

---

#### 需求 2.2：批量分析功能

**用户故事**：作为用户，我希望能够批量分析多个股票，以便高效地筛选投资标的。

**验收标准**：

2.2.1 当用户提交批量分析请求时，系统应创建多个分析任务并加入任务队列。

2.2.2 当批量任务使用公共模型时，系统应限制最多提交 5 个任务，超过部分需等待前序任务完成。

2.2.3 当批量任务使用用户自定义模型时，系统应不限制任务数量，按提交顺序执行。

2.2.4 当批量任务执行时，系统应在任务列表页面显示每个任务的进度和状态。

2.2.5 当用户查看批量任务详情时，系统应显示当前执行阶段和已完成的报告。

2.2.6 当用户取消批量任务中的单个任务时，系统应仅取消该任务，不影响其他任务。

2.2.7 当用户取消整个批量任务时，系统应取消所有未开始的任务，保留已完成的结果。

2.2.8 当批量任务全部完成时，系统应生成批量分析汇总报告。

---

#### 需求 2.3：公共模型资源池管理

**用户故事**：作为系统，我希望能够管理公共 AI 模型的并发资源池，以便公平分配资源给所有用户。

**验收标准**：

2.3.1 当管理员配置公共模型时，系统应设置模型的总并发上限（如 10 个）。

2.3.2 当用户使用公共模型执行任务时，系统应限制每个用户最多占用 1 个并发槽位。

2.3.3 当公共模型槽位已满时，系统应将新任务加入等待队列，按提交时间先后顺序排队。

2.3.4 当队列中的任务等待时，系统应显示排队位置（前面还有几个任务）。

2.3.5 当用户的任务完成时，系统应释放该用户占用的槽位，分配给队列中的下一个任务。

2.3.6 当用户使用自定义模型时，系统应不占用公共模型槽位，独立计算并发。

2.3.7 当不同阶段使用不同 AI 模型时，系统应在阶段完成后释放前一阶段的模型资源。

2.3.8 当用户在同一时间提交多个使用公共模型的任务时，系统应只执行第一个任务，其他任务等待。

---

#### 需求 2.4：并发控制

**用户故事**：作为系统，我希望能够控制系统并发资源使用，以便确保系统稳定运行和资源合理分配。

**验收标准**：

2.4.1 当 AI 模型设置并发限制为 N 时，系统应确保该模型同时进行的 API 请求不超过 N 个。

2.4.2 当单个任务智能体并发设置为 M 时，系统应确保单个分析任务同时执行的智能体不超过 M 个。

2.4.3 当单任务并发 M 大于模型并发 N 时，系统应实际并发数限制为 N。

2.4.4 当多个任务同时请求同一模型时，系统应共享该模型的并发配额。

2.4.5 当并发配额耗尽时，系统应将新请求加入等待队列。

2.4.6 当等待队列中的请求超时（5 分钟）时，系统应标记该请求失败并通知用户。

2.4.7 当用户配置批量任务并发数时，系统应控制同时执行的分析任务数量。

---

#### 需求 2.5：分析报告管理

**用户故事**：作为用户，我希望能够管理和查询分析报告，以便回顾历史分析结果和筛选投资标的。

**验收标准**：

2.5.1 当智能体生成报告时，系统应存储报告内容、推荐结果（买入/卖出/持有）、买入价格、卖出价格、生成时间、关联任务 ID 和用户 ID。

2.5.2 当用户查询历史报告时，系统应支持按股票代码、时间范围、任务状态筛选。

2.5.3 当用户筛选报告时，系统应支持按推荐结果（买入/卖出/持有）筛选。

2.5.4 当用户查看报告详情时，系统应显示所有阶段的完整报告内容。

2.5.5 当用户查看部分完成的任务报告时，系统应显示已完成阶段的报告和未完成阶段的状态。

2.5.6 当管理员查询报告时，系统应允许查看所有用户的报告并显示用户信息。

2.5.7 当普通用户查询报告时，系统应仅返回该用户自己的报告。

2.5.8 当用户按股票排序报告列表时，系统应支持按股票代码升序或降序排列。

---

#### 需求 2.6：任务中心

**用户故事**：作为用户，我希望能够管理分析任务，以便跟踪任务进度和处理异常情况。

**验收标准**：

2.6.1 当用户访问任务中心时，系统应显示任务列表，包含任务 ID、股票代码、推荐结果、状态、创建时间和进度。

2.6.2 当任务状态变化时，系统应实时更新任务列表显示。

2.6.3 当用户筛选任务时，系统应支持按状态（进行中/已完成/失败/已取消）筛选。

2.6.4 当用户查看进行中任务时，系统应显示当前执行阶段和智能体状态。

2.6.5 当用户取消进行中任务时，系统应停止任务执行并更新状态为已取消。

2.6.6 当用户重试失败任务时，系统应创建新任务并使用相同的分析参数。

2.6.7 当用户删除任务记录时，系统应同时删除关联的报告数据。

2.6.8 当管理员访问任务中心时，系统应显示所有用户的任务并支持按用户筛选。

### 三、高级功能模块

#### 需求 3.1：阶段开关控制

**用户故事**：作为用户，我希望能够控制分析流程的阶段，以便根据需求选择执行完整或简化的分析流程。

**验收标准**：

3.1.1 当用户配置阶段开关时，系统应允许独立启用或禁用第二、三、四阶段。

3.1.2 当第二阶段禁用时，系统应跳过研究员辩论直接进入下一启用阶段。

3.1.3 当第三阶段禁用时，系统应跳过风险评估直接进入下一启用阶段。

3.1.4 当第四阶段禁用时，系统应不生成总结报告，以各阶段报告作为最终结果。

3.1.5 当所有可选阶段禁用时，系统应仅执行第一阶段分析师团队。

3.1.6 当用户为单次任务指定阶段开关时，系统应使用任务级配置覆盖用户默认配置。

3.1.7 当阶段被禁用时，后置阶段智能体应基于可用数据生成报告，而非报错。

---

#### 需求 3.2：实时状态推送

**用户故事**：作为用户，我希望能够实时查看分析过程，以便了解分析进度和智能体工作状态。

**验收标准**：

3.2.1 当用户打开分析详情页面时，系统应建立 WebSocket 连接。

3.2.2 当用户关闭分析详情页面或切换到其他页面时，系统应断开 WebSocket 连接。

3.2.3 当智能体开始执行时，系统应推送智能体名称和开始时间。

3.2.4 当智能体调用工具时，系统应推送工具名称、输入参数和调用状态。

3.2.5 当工具调用完成时，系统应推送工具返回结果摘要和耗时。

3.2.6 当智能体生成报告时，系统应推送报告生成完成事件和报告摘要。

3.2.7 当阶段切换时，系统应推送阶段切换事件和新阶段信息。

3.2.8 当任务完成或失败时，系统应推送最终状态。

3.2.9 当 WebSocket 连接断开时，系统应支持客户端重新连接并恢复状态推送。

3.2.10 当单用户 WebSocket 连接数超过 5 个时，系统应断开最早的连接。

---

#### 需求 3.3：权限与多用户隔离

**用户故事**：作为系统管理员，我希望能够管理用户权限和数据隔离，以便确保系统安全和用户数据隐私。

**验收标准**：

3.3.1 当超级管理员配置系统资源时，系统应允许管理系统级 AI 模型、MCP 服务器和默认智能体模板。

3.3.2 当普通管理员访问系统时，系统应允许查看所有用户的任务和报告。

3.3.3 当普通用户访问系统时，系统应仅允许访问自己的配置、任务和报告。

3.3.4 当用户使用公共 AI 模型时，系统应共享模型并发配额，每个用户最多占用 1 个槽位。

3.3.5 当用户使用个人 AI 模型时，系统应使用用户独立的并发配额。

3.3.6 当用户查询数据时，系统应自动添加用户 ID 过滤条件。

3.3.7 当管理员查询数据时，系统应允许选择是否包含用户过滤条件。

3.3.8 当用户被禁用时，系统应停止该用户进行中的任务，不允许创建新任务。

---

#### 需求 3.4：工具管理与可用性检测

**用户故事**：作为用户，我希望能够管理智能体可用的工具，以便确保分析过程使用正确的数据源。

**验收标准**：

3.4.1 当系统启动时，系统应检测所有配置的 MCP 服务器可用性。

3.4.2 当 MCP 服务器不可用时，系统应自动禁用该服务器并记录日志。

3.4.3 当智能体执行前，系统应获取当前可用的工具列表。

3.4.4 当智能体连续 3 次调用同一工具且参数完全相同时，系统应判定为陷入循环。

3.4.5 当检测到工具循环时，系统应自动禁用该工具并通知智能体"工具 X 已被禁用，请尝试其他方法"。

3.4.6 当工具调用失败超过阈值（3 次）时，系统应自动禁用该工具并通知用户。

3.4.7 当用户手动检测工具可用性时，系统应执行连接测试并更新状态。

3.4.8 当用户为智能体配置工具时，系统应仅显示当前可用的工具列表。

3.4.9 当本地工具接口预留时，系统应提供本地工具注册和调用的扩展接口。

#### 需求 3.5：辩论轮次控制

**用户故事**：作为用户，我希望能够控制辩论轮次，以便平衡分析深度和执行时间。

**验收标准**：

3.5.1 当用户配置第二阶段辩论轮次时，系统应控制看涨和看跌研究员的辩论次数。

3.5.2 当用户配置第三阶段讨论轮次时，系统应控制风险分析师的讨论次数。

3.5.3 当辩论达到最大轮次时，系统应自动进入裁决阶段。

3.5.4 当辩论轮次设置为 0 时，系统应跳过辩论直接进入裁决。

3.5.5 当用户为单次任务指定辩论轮次时，系统应使用任务级配置覆盖用户默认配置。

#### 需求 3.6：错误处理与恢复

**用户故事**：作为用户，我希望系统能够处理各种错误情况，以便获得稳定可靠的分析服务。

**验收标准**：

3.6.1 当 AI 模型 API 调用失败时，系统应重试最多 3 次，每次间隔递增（1 秒、2 秒、4 秒）。

3.6.2 当重试后仍然失败时，系统应标记智能体执行失败并记录错误详情。

3.6.3 当单个智能体失败时，系统应继续执行其他智能体，不中断整个任务。

3.6.4 当关键智能体失败时，系统应标记任务为部分完成并保留已有报告。

3.6.5 当工具调用超时时，系统应返回超时错误并允许智能体继续执行。

3.6.6 当系统重启时，系统应恢复进行中任务的状态并继续执行。

3.6.7 当任务恢复时，系统应从当前进度的智能体继续执行，已完成的报告保留。

3.6.8 当任务恢复时发现配置中的智能体已被删除时，系统应标记任务失败并提示"任务恢复失败：配置的智能体已被删除"。

3.6.9 当数据库连接失败时，系统应暂停新任务接收并尝试重连。

3.6.10 当单个智能体执行超时时，系统应标记该智能体失败，保留其他已完成智能体的报告。

---

#### 需求 3.7：配置持久化

**用户故事**：作为用户，我希望配置能够持久保存，以便在不同会话中使用相同的配置。

**验收标准**：

3.7.1 当系统默认智能体模板更新时，系统应存储在 YAML 配置文件中。

3.7.2 当用户保存个人配置时，系统应存储到 MongoDB 用户配置集合。

3.7.3 当用户登录时，系统应加载用户个人配置。

3.7.4 当用户配置不存在时，系统应从系统默认模板初始化。

3.7.5 当配置格式升级时，系统应自动迁移旧格式配置到新格式，不丢失用户数据。

3.7.6 当用户导出配置时，系统应生成 JSON 格式的配置文件。

3.7.7 当用户导入配置时，系统应验证配置格式并合并到现有配置。

---

#### 需求 3.8：API Key 存储

**用户故事**：作为系统，我希望能够安全存储 API Key，以便用户无需重复输入。

**验收标准**：

3.8.1 当用户保存 AI 模型配置时，系统应将 API Key 加密存储到数据库（使用应用层加密，密钥由环境变量提供）。

3.8.2 当系统记录日志时，系统应将 API Key 替换为 [REDACTED] 不记录明文。

3.8.3 当 API Key 返回给前端时，系统应只显示脱敏后的部分（如 sk-****1234）。

---

#### 需求 3.9：输入验证

**用户故事**：作为系统，我希望能够验证用户输入，以便防止恶意输入和数据库注入。

**验收标准**：

3.9.1 当用户配置智能体提示词时，系统应限制最大长度为 10000 个字符。

3.9.2 当用户输入包含 NoSQL 注入特征（如 $where, $ne 等）时，系统应拒绝该输入或进行清洗。

3.9.3 当用户输入包含 XSS 特征时，系统应进行 HTML 转义处理。

3.9.4 当用户提交的参数类型不匹配时，系统应返回参数类型错误提示。

### 四、数据模型需求

#### 需求 4.1：分析任务数据模型

**用户故事**：作为系统，我需要定义分析任务的数据结构，以便存储和管理任务信息。

**验收标准**：

4.1.2 当推荐结果字段存储时，系统应使用枚举值："买入"、"卖出"、"持有"。

4.1.3 当买入价格和卖出价格存储时，系统应使用浮点数类型，可为空。

4.1.4 当配置快照存储时，系统应保存任务创建时的完整配置，以便后续任务恢复。

4.1.5 当任务状态存储时，系统应使用标准枚举：PENDING（待执行）、RUNNING（执行中）、COMPLETED（已完成）、FAILED（失败）、CANCELLED（已取消）、STOPPED（已停止）、EXPIRED（已过期）。
