# 多用户架构设计文档

## 1. 设计目标

构建一个安全、隔离、可扩展的多用户管理系统，支持：
1.  **千万级用户扩展**：基于 MongoDB 的分布式存储设计。
2.  **严格的资源隔离**：确保用户 A 无法访问用户 B 的配置、数据和文件。
3.  **灵活的权限控制**：支持自定义角色和细粒度权限配置。
4.  **混合配置模式**：支持“系统默认配置”与“用户自定义配置”的叠加与覆盖。

## 3. 详细模块设计

### 3.1 身份系统升级

#### 3.1.1 用户唯一标识
*   **方案**：采用 MongoDB `ObjectId` 作为内部主键，对外暴露时序列化为字符串。
*   **辅助标识**：`username` (唯一索引), `email` (唯一索引), `phone` (可选，唯一索引)。

#### 3.1.2 数据模型变更 (`User`)
```python
class User(BaseModel):
    id: PyObjectId
    username: str
    email: str
    hashed_password: str  # 升级为 bcrypt
    
    # 权限关联
    role_ids: List[str] = ["user"]  # 关联 Roles 集合
    
    # 状态
    status: str = "active"  # active, suspended, pending
    
    # 隔离空间
    namespace_id: str  # 默认为 user_id，用于逻辑隔离
    
    # 审计
    created_at: datetime
    last_login_ip: str
```

### 3.2 配置隔离系统 (`UserConfig`)

为了解决全局配置问题，引入**分层配置架构**：

1.  **System Level**: 管理员配置的默认 API Key（用于免费试用或公共服务）。
2.  **User Level**: 用户配置的私有 API Key（用于高频使用或隐私保护）。

**数据模型设计**:
```python
class UserLLMConfig(BaseModel):
    """用户私有 LLM 配置"""
    user_id: PyObjectId  # 索引
    provider: str        # e.g., "openai"
    api_key: str         # AES-256 加密存储
    api_base: Optional[str]
    is_enabled: bool = True
    
    class Config:
        collection = "user_llm_configs"
```

**配置读取逻辑**:
```python
async def get_provider_config(user_id, provider_name):
    # 1. 尝试获取用户私有配置
    user_config = await db.user_llm_configs.find_one({"user_id": user_id, "provider": provider_name})
    if user_config and user_config.is_enabled:
        return decrypt(user_config.api_key)
        
    # 2. 如果没有私有配置，检查用户是否有权使用系统配置
    if check_permission(user_id, "use_system_llm"):
        return get_system_config(provider_name)
        
    raise ConfigNotFoundError("未找到可用的 API 配置")
```

### 3.3 资源配额管理 (`QuotaManager`)

基于 Redis 实现实时的配额控制：

*   **Key 设计**: `quota:{user_id}:{resource_type}:{date}`
*   **资源类型**:
    *   `analysis_count`: 每日分析次数
    *   `token_usage`: 每日 Token 消耗
    *   `concurrent_tasks`: 当前并发任务数
*   **操作流程**:
    1.  请求进来 -> 检查 Redis 计数器。
    2.  如果 `< limit` -> INCR -> 放行。
    3.  如果 `>= limit` -> 拒绝 (HTTP 429)。

## 4. API 接口改造规划

| 接口路径 | 方法 | 改造内容 |
| :--- | :--- | :--- |
| `/api/auth/login` | POST | 增加最后登录 IP 和时间记录，返回角色信息 |
| `/api/users/me/config` | GET | **新增**：获取当前用户的私有配置列表（脱敏） |
| `/api/users/me/config` | POST | **新增**：添加/更新私有配置（自动加密 Key） |
| `/api/analysis/submit` | POST | 在 Service 层注入 `user_id`，根据 ID 加载对应配置 |
| `/api/admin/users` | GET | **新增**：管理员用户列表查询（分页、搜索） |
| `/api/admin/users/{id}/quota` | PUT | **新增**：动态调整用户配额 |

## 5. 迁移策略

1.  **数据迁移**：编写脚本将现有 `User` 表中的 `is_admin=True` 转换为关联 `admin` 角色。
2.  **配置迁移**：保留现有的 `config` 集合作为系统级配置，新建 `user_llm_configs` 集合。
3.  **兼容性**：在过渡期，如果用户没有私有配置，默认回退到系统配置，保证服务不中断。
