# 数据隔离方案说明

## 1. 概述

在多用户环境中，数据隔离是保障数据安全和隐私的底线。本方案采用**逻辑隔离**（基于字段）与**物理隔离**（基于目录）相结合的方式，确保用户数据互不可见。

## 2. 数据库隔离方案 (MongoDB)

### 2.1 核心原则
*   **所有业务集合必须包含 `user_id` 字段**。
*   **所有查询必须强制带上 `user_id` 过滤条件**。
*   **建立 `user_id` 索引**以优化查询性能。

### 2.2 集合改造清单

| 集合名称 | 改造措施 | 索引策略 | 备注 |
| :--- | :--- | :--- | :--- |
| `analysis_tasks` | 确保 `user_id` 存在且非空 | `create_index([("user_id", 1), ("created_at", -1)])` | 任务列表查询高频 |
| `analysis_results` | 增加 `user_id` 字段 | `create_index([("user_id", 1), ("task_id", 1)])` | 结果归属验证 |
| `favorites` | 从 `User` 嵌入拆分为独立集合 | `create_index([("user_id", 1), ("stock_code", 1)], unique=True)` | 支持大量自选股 |
| `paper_accounts` | 维持现状 (`user_id` 已存在) | `create_index("user_id", unique=True)` | 账户唯一性 |
| `operation_logs` | 增加 `user_id` 字段 | `create_index([("user_id", 1), ("timestamp", -1)])` | 用户审计日志 |

### 2.3 代码层拦截 (Middleware/Service)

在 `BaseService` 或数据访问层（DAO）实现自动注入：

```python
class BaseRepository:
    def __init__(self, user_id: str):
        self.user_id = user_id
        
    def find(self, query: dict):
        # 强制注入 user_id，防止越权
        safe_query = {**query, "user_id": self.user_id}
        return self.collection.find(safe_query)
```

## 3. 文件存储隔离方案

### 3.1 目录结构规范

目前系统文件存储较为扁平，需重构为以下树状结构：

```
data/
├── system/              # 系统级公共文件
│   ├── models/          # 本地模型文件
│   └── cache/           # 公共行情缓存 (所有用户共享)
├── users/               # 用户私有空间
│   ├── {user_id_hash}/  # 使用 Hash 避免直接暴露 ID
│   │   ├── reports/     # 分析报告 (Markdown/PDF)
│   │   ├── temp/        # 临时文件
│   │   └── exports/     # 导出数据
│   └── ...
└── logs/                # 系统日志
```

### 3.2 路径生成算法

```python
import hashlib
import os

def get_user_storage_path(user_id: str) -> str:
    # 使用 SHA256 前 8 位作为目录名，避免目录遍历攻击
    dir_name = hashlib.sha256(user_id.encode()).hexdigest()[:16]
    path = os.path.join("data", "users", dir_name)
    os.makedirs(path, exist_ok=True)
    return path
```

### 3.3 访问控制

*   **静态资源服务**：Nginx 或 FastAPI StaticFiles 挂载时，不能直接挂载根目录 `data/users`。
*   **下载接口**：必须通过应用程序接口 `/api/files/download/{file_id}` 访问。
    *   接口逻辑：
        1.  验证 Token。
        2.  根据 `file_id` 查库获取 `file_path` 和 `owner_user_id`。
        3.  校验 `current_user.id == owner_user_id`。
        4.  使用 `FileResponse` 返回文件流。

## 4. 缓存隔离 (Redis)

*   **Session/Token**: `auth:token:{token_hash}` -> `user_info`
*   **用户锁**: `lock:user:{user_id}:{resource}`
*   **公共数据**: `market:quote:{stock_code}` (不带 user_id，因为行情是公共的)
*   **私有缓存**: `user:{user_id}:analysis:progress:{task_id}`

**注意**：行情数据（Market Data）属于**公共读**数据，不需要隔离，应由所有用户共享缓存，以节省内存和带宽。
