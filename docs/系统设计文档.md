## 一、系统概述

### 1.1 系统定位

**StockAnalysis** 是一个基于多智能体技术的股票分析平台,采用 **模块化单体 (Modular Monolith)** 架构,支持多用户认证、完全的数据隔离,并提供智能化的投资分析和建议。

**核心价值**:
- 🤖 **AI 驱动**: 基于 LangGraph 的多阶段智能体协作分析
- 👥 **多租户支持**: 完善的用户数据隔离和权限管理
- 🔧 **高度可配置**: 用户可自定义 AI 模型、MCP 工具、智能体行为
- 📊 **实时反馈**: WebSocket 推送任务进度和中间结果
- 🔒 **安全可靠**: RBAC 权限控制、数据加密、审计日志

### 1.2 技术栈总览

| 层次 | 技术选型 | 用途 |
|------|---------|------|
| **前端** | Vue 3 + TypeScript + Vite | 现代化响应式界面 |
| **UI 组件** | Element Plus | 企业级 UI 组件库 |
| **状态管理** | Pinia | 轻量级状态管理 |
| **后端框架** | FastAPI | 高性能异步 Web 框架 |
| **工作流引擎** | LangGraph | 多智能体协作编排 |
| **LLM 接入** | OpenAI 兼容接口 | 统一模型接入层 |
| **工具协议** | MCP (Model Context Protocol) | 标准化工具调用 |
| **数据库** | MongoDB | 文档型数据存储 |
| **缓存** | Redis | 会话、队列、并发控制 |
| **认证** | JWT (JSON Web Token) | 无状态身份认证 |
| **容器化** | Docker + Docker Compose | 容器化部署 |

### 2.2 模块化单体架构

**设计理念**: 将单体应用按业务领域划分为清晰的模块,每个模块有独立的职责和边界,同时保持部署和运维的简单性。

#### 模块划分原则

| 模块类型 | 定义 | 示例 |
|---------|------|------|
| **核心模块** (core/) | 系统必需的基础功能,不可随意修改 | 认证、用户、RBAC、设置 |
| **业务模块** (modules/) | 业务功能,可独立开发、测试、扩展 | TradingAgents、选股、问股 |
| **前端核心** (core/) | 前端基础框架,通用组件 | 布局、路由、HTTP 客户端 |
| **前端业务** (modules/) | 前端业务页面和逻辑 | 分析页面、设置页面 |

#### 模块交互规范

```
核心规则:
1. 核心模块之间 → 尽量减少依赖,通过接口通信
2. 业务模块 → 只依赖核心模块,不依赖其他业务模块
3. 前端业务模块 → 通过 API 与后端通信,不直接访问数据层
4. 跨模块调用 → 通过明确的 API 接口,不共享内部实现

交互模式:
┌─────────────┐      API       ┌─────────────┐
│ 业务模块 A   │ ──────────────→│ 业务模块 B   │
│  (Trading)  │                │  (Screener) │
└─────────────┘                └─────────────┘
       ↓                              ↓
┌─────────────┐      API       ┌─────────────┐
│  核心模块    │ ──────────────→│  核心模块    │
│  (用户)      │                │  (认证)      │
└─────────────┘                └─────────────┘
```

## 三、核心模块设计

### 3.1 认证与授权模块 (core/auth/)

#### 3.1.1 JWT 认证流程

```
用户登录流程:
┌──────────┐    POST /api/users/login    ┌──────────┐
│  前端应用 │ ──────────────────────────→│  后端API  │
└──────────┘                              └──────────┘
     │                                            │
     │ 1. 提交 username + password + captcha      │
     │                                            │
     │                                    2. 验证用户名密码
     │                                    3. 检查验证码
     │                                    4. 检查账号状态
     │                                    5. 检查登录失败次数
     │                                    6. 生成 JWT Token
     │                                            │
     │ ←─────────────────────────────────────│
     │  返回 access_token + refresh_token       │
     │                                            │
     7. 存储 Token 到 localStorage              │
     8. 后续请求携带 Authorization: Bearer <token> │
```

#### 3.1.2 Token 机制

| Token 类型 | 有效期 | 用途 | 存储位置 |
|-----------|--------|------|---------|
| **Access Token** | 30分钟 | 访问受保护资源 | LocalStorage |
| **Refresh Token** | 7天 | 刷新 Access Token | HttpOnly Cookie |

#### 3.1.3 RBAC 权限系统

**角色定义**:

```
角色层级:
GUEST (访客)
    ↓ 注册后变为
USER (普通用户)
    ↓ 管理员提升为
ADMIN (管理员)
    ↓ 超级管理员提升为
SUPER_ADMIN (超级管理员)
```

**权限粒度**:
- 模块级权限: `trading_agents:read`, `trading_agents:write`
- 资源级权限: `user:read`, `user:create`, `user:update`, `user:delete`
- 系统级权限: `system:config`, `audit:read`

**权限检查流程**:
```
用户请求 → JWT 解析 → 获取 user_id → 查询用户角色 → 检查权限
    ↓
权限通过 → 执行业务逻辑
权限拒绝 → 返回 403 Forbidden
```

### 3.2 用户管理模块 (core/user/)

#### 3.2.1 用户状态机

```
注册 → PENDING (待审核)
       ↓
    管理员审核
       ↓
    ├─ APPROVE → ACTIVE (已激活,可登录)
    ├─ REJECT → REJECTED (已拒绝,数据保留但不可登录)
    └─ 禁用 → DISABLED (暂时禁止登录)
```

#### 3.2.2 用户数据隔离 (强制执行)

**隔离原则**:
- **MongoDB**: 所有用户相关的文档 **必须** 包含 `user_id` 字段
- **Redis**: 使用带命名空间的键 (`user:{user_id}:...`)
- **Service 层**: 所有查询和操作 **必须** 强制过滤 `user_id`

**示例**:
```python
# ❌ 错误: 直接查询全表
async def get_all_tasks():
    return await db.tasks.find({})

# ✅ 正确: 强制过滤 user_id
async def get_user_tasks(user_id: str):
    return await db.tasks.find({"user_id": user_id})
```

### 3.5 安全机制模块 (core/security/)

#### 3.5.1 滑块验证码

**触发条件**:
- 登录失败次数超过阈值 (默认 3 次)
- IP 地址不在信任列表中

**验证流程**:
```
前端 → 生成滑块图片 → 用户拖动滑块 → 验证位置 → 提交 token
        ↓
后端 → 验证 token → 验证通过 → 允许登录/注册
```

#### 3.5.2 IP 信任机制

**工作原理**:
- 用户成功登录后,记录 IP 地址
- 同一 IP 多次成功登录 (默认 5 次),标记为信任 IP
- 信任 IP 跳过验证码检查

#### 3.5.3 登录安全限流

| 限流类型 | 配置 | 说明 |
|---------|------|------|
| **最大失败次数** | 5次 | 超过后账号锁定 30 分钟 |
| **失败窗口期** | 5分钟 | 统计失败次数的时间窗口 |
| **锁定时长** | 30分钟 | 账号锁定时间 |
| **IP 信任阈值** | 5次 | 标记为信任 IP 的成功登录次数 |

## 四、业务模块设计

### 4.1 TradingAgents 智能分析系统

这是本系统的核心业务模块,基于 LangGraph 实现多智能体协作的股票分析工作流。

#### 4.1.1 四阶段工作流

```
┌─────────────────────────────────────────────────────────────────┐
│                    TradingAgents 工作流                          │
└─────────────────────────────────────────────────────────────────┘

阶段1: 分析师团队 (Individual Analysis)
├── 并行执行 3-4 个分析师
│   ├── 技术面分析师 (MACD, RSI, KDJ)
│   ├── 基本面分析师 (财务报表, PE, PB)
│   ├── 市场情绪分析师 (新闻情感, 社交媒体)
│   └── 自定义分析师 (可扩展)
└── 输出: 多份独立分析报告

         ↓

阶段2: 研究员辩论 (Research & Debate) [可选]
├── 看涨研究员 vs 看跌研究员
├── 多轮交叉辩论 (1-3轮)
├── 研究经理裁决
└── 交易员制定投资计划
    输出: 辩论总结 + 投资计划

         ↓

阶段3: 风险评估 (Risk Assessment) [可选]
├── 激进派观点
├── 保守派观点
├── 中性派观点
└── 首席风控官总结
    输出: 风险评估报告

         ↓

阶段4: 总结输出 (Summary) [可选]
└── 综合所有信息生成最终报告
    输出: 买入/卖出/持有建议 + 目标价位
```

#### 4.1.2 核心组件

**1. 任务管理器 (task_manager.py)**
- 创建分析任务 (单股/批量)
- 任务状态管理 (pending → running → completed/failed)
- 任务恢复机制 (系统重启后恢复运行中任务)
- 任务过期处理 (24小时未完成自动过期)

**2. 并发控制器 (concurrency_controller.py)**
- 公共模型资源池管理 (每个用户最多占 1 个槽位)
- 用户级并发限制 (用户可同时运行的任务数)
- 批量任务并发控制 (批量任务的子任务并发数)
- FIFO 队列和优先级队列

**3. MCP 工具适配器 (mcp_adapter.py)**
- MCP 协议适配 (stdio/SSE/HTTP)
- 工具注册与发现
- 并发调用控制 (个人 100/用户,公共 10/用户)
- 健康检查与自动重连

**4. 工具循环检测器 (loop_detector.py)**
- 检测连续重复的工具调用 (阈值: 3次)
- 自动禁用问题工具
- 防止 AI 陷入死循环

**5. WebSocket 管理器 (websocket/manager.py)**
- 按需连接 (用户打开详情页时连接)
- 实时推送任务进度
- 心跳保活机制
- 最多 5 个并发连接/用户

#### 4.1.3 配置优先级规则

```
三层配置优先级:
1. 用户自定义配置 (is_customized=True) - 最高优先级
2. 系统公共配置 (user_id="system_public")
3. 默认 YAML 模板 (从代码库加载)

智能体配置包含:
├── enabled: 是否启用该智能体

```

#### 4.1.4 双模型配置

### 12.1 模块扩展

**新增业务模块步骤**:
```
1. 在 backend/modules/ 创建新目录
2. 实现 api.py、service.py、schemas.py
3. 在 main.py 中注册路由
4. 在 frontend/src/modules/ 创建对应前端模块
5. 在 module_loader.ts 中添加路由
```

### 12.2 智能体扩展

**新增自定义智能体**:
```
1. 编辑 agents.yaml 配置文件
2. 定义智能体:
   ├── slug (唯一标识)
   ├── name (显示名称)
   ├── role_definition (系统提示词)
   └── enabled_mcp_servers (启用的 MCP 服务器)
3. 工厂模式自动创建智能体实例
```

### 12.3 MCP 工具扩展

**集成新 MCP 服务器**:
```
1. 在 MCP 服务器管理页面创建配置
2. 选择传输模式 (stdio/SSE/HTTP)
3. 配置连接参数
4. 系统自动:
   ├── 连接测试
   ├── 工具发现
   └── 注册到工具注册表
5. 智能体配置中启用该 MCP 服务器
```

### 12.4 多语言支持 (预留)

**国际化架构**:
```
i18n/
├── locales/
│   ├── zh-CN.json (简体中文)
│   ├── en-US.json (英文)
│   └── zh-TW.json (繁体中文)
└── index.ts
```
