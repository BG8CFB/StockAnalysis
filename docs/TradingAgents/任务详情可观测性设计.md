## 一、业务需求分析

### 1.1 用户场景

| 场景 | 用户需求 | 期望看到的信息 |
|------|---------|--------------|
| **执行中监控** | 任务正在运行，想了解进度 | 当前执行到哪一步、哪个智能体在工作、大概还需要多久 |
| **调试分析** | 分析结果不符合预期，想查原因 | 智能体调用了哪些工具、获取了什么数据、推理过程是什么 |
| **成本追踪** | 想了解 API 调用成本 | 每个 LLM 调用的 token 消耗、估算成本 |
| **事后回顾** | 任务完成后复盘 | 整个执行流程的时间线、各阶段耗时、瓶颈在哪里 |
| **并发理解** | 想理解智能体的执行逻辑 | 哪些智能体是并发执行的、哪些是串行的、为什么这样设计 |

### 1.2 信息分层

根据"是否需要持久化"将信息分为三层：

| 层级 | 内容 | 存储策略 | 理由 |
|------|------|---------|------|
| **L1 - 核心结果** | 最终报告、推荐结论、价格建议 | 永久存储 | 用户需要长期查阅 |
| **L2 - 执行摘要** | 阶段记录、工具统计、Token 消耗 | 任务完成后保留 7-30 天 | 用于调试和优化，但长期占用存储 |
| **L3 - 实时过程** | LLM 对话流、思考过程、实时工具输出 | **不存储，仅实时推送** | 数据量大、回看价值低、刷新后重新生成 |

### 1.3 并发展示的特殊挑战

当前系统有**并发执行的智能体**（如阶段 1 的多个分析师并发），这带来了几个设计问题：

1. **时间线如何表示**：传统的线性时间线无法表达"同时进行"
2. **状态如何同步**：多个智能体同时运行，用户如何知道当前谁在做什么
3. **详情如何展示**：用户可能同时关注多个智能体，还是只能看一个
4. **进度如何计算**：并发时多个智能体同时推进，整体进度如何计算

**解决方案预览**：
- 用"泳道图"或"分组卡片"表示并发
- 并发智能体横向排列，串行智能体纵向排列
- 一次只能展开一个智能体的详情，避免混乱
- 进度按"已完成智能体/总智能体"计算

## 二、数据模型设计

### 2.1 任务文档结构

现有 `analysis_tasks` 文档需要扩展以下字段：

#### 2.1.1 工具调用统计

**用途**：快速查询每个智能体使用了哪些工具，无需遍历详细记录

**结构**：
```
tool_stats: {
  "<agent_slug>": {
    "total_calls": 5,              // 该智能体总共调用工具次数
    "total_duration_ms": 1500,     // 总耗时
    "tools": {
      "<tool_name>": {
        "count": 2,                // 该工具调用次数
        "success_count": 2,        // 成功次数
        "fail_count": 0,           // 失败次数
        "total_duration_ms": 500   // 该工具总耗时
      }
    }
  }
}
```

**更新时机**：每次工具调用完成时，异步更新统计

**查询场景**：
- 任务详情页展示工具调用统计（按智能体/按工具分组）
- 快速了解哪些工具被频繁调用
- 识别响应慢的工具

#### 2.1.2 阶段执行记录

**用途**：记录每个阶段的执行过程，用于前端绘制时间线

**结构**：
```
phase_executions: [
  {
    "phase": 1,
    "phase_name": "分析师团队",
    "started_at": "2024-01-01T10:00:00Z",
    "completed_at": "2024-01-01T10:05:00Z",
    "execution_mode": "concurrent",  // "concurrent" 或 "serial"
    "max_concurrency": 3,
    "agents": [
      {
        "slug": "technical_analyst",
        "name": "技术分析师",
        "started_at": "2024-01-01T10:00:00Z",
        "completed_at": "2024-01-01T10:02:00Z",
        "status": "completed",  // pending, running, completed, failed
        "token_usage": {
          "prompt_tokens": 1000,
          "completion_tokens": 500,
          "total_tokens": 1500
        },
        "position": 1  // 在并发组中的位置（用于前端排序）
      },
      // ... 其他智能体
    ]
  },
  // ... 其他阶段
]
```

**更新时机**：
- 阶段开始时：创建记录，设置 `started_at`、`execution_mode`、`agents` 列表
- 智能体开始时：更新对应 agent 的 `status` 为 `running`，设置 `started_at`
- 智能体完成时：更新对应 agent 的 `status` 为 `completed`，设置 `completed_at`、`token_usage`
- 阶段完成时：设置 `completed_at`

**并发相关的字段**：
- `execution_mode`：告诉前端这是并发还是串行
- `max_concurrency`：最大并发数（仅并发模式有效）
- `position`：智能体在并发组中的位置（前端用于排序和布局）

#### 2.1.3 并发组信息

**用途**：告诉前端哪些智能体是并发执行的，便于 UI 分组展示

**结构**：
```
concurrency_groups: [
  {
    "group_id": "phase1_group1",
    "phase": 1,
    "agents": ["technical_analyst", "fundamental_analyst", "sentiment_analyst"],
    "max_concurrency": 3,
    "created_at": "2024-01-01T10:00:00Z"
  },
  {
    "group_id": "phase3_group1",
    "phase": 3,
    "agents": ["aggressive_risk_analyst", "conservative_risk_analyst", "neutral_risk_analyst"],
    "max_concurrency": 3,
    "created_at": "2024-01-01T10:05:00Z"
  }
]
```

**创建时机**：任务启动时，根据智能体配置预先生成

**并发规则**：
- **阶段 1**：用户选择的智能体（最多 3 个）并发执行
- **阶段 2**：看多/看空辩手串行执行（交替发言）
- **阶段 3**：激进/保守/中性三方并发执行
- **阶段 4**：总结智能体单独执行

### 2.2 工具调用追踪文档

**用途**：详细记录每次工具调用的输入输出，用于调试和问题排查

**文档结构**：
```
{
  "_id": ObjectId,
  "task_id": "xxx",
  "user_id": "xxx",
  "phase": 1,
  "agent_slug": "technical_analyst",
  "agent_name": "技术分析师",

  // 工具信息
  "tool_name": "get_stock_price",
  "tool_input": { "stock_code": "000001", "market": "a_share" },
  "tool_output": { "price": 10.50, "change": "+2.3%" },
  "success": true,
  "error": null,

  // 时间信息
  "started_at": "2024-01-01T10:00:30Z",
  "completed_at": "2024-01-01T10:00:31Z",
  "duration_ms": 500,

  // 元数据
  "timestamp": "2024-01-01T10:00:31Z",
  "trace_id": "uuid",  // 用于关联和追踪
  "concurrent_group": "phase1_group1"  // 所属并发组
}
```

**写入时机**：工具调用完成后立即写入（异步，不阻塞主流程）

**查询场景**：
- 用户点击某个工具调用记录，想看详细的输入输出
- 调试某个失败的工具调用
- 统计工具性能（响应时间、成功率）

### 2.3 数据保留策略

| 数据类型 | 保留时间 | 清理方式 | 清理时机 |
|---------|---------|---------|---------|
| 任务文档（含最终报告） | 永久 | 不清理 | - |
| 阶段执行记录、工具统计 | 30 天 | 从任务文档中删除字段 | 每天凌晨 3 点 |
| 工具调用追踪 (agent_traces) | 7 天 | 删除记录 | 每天凌晨 3 点 |
| LLM 消息流 | 不保留 | 不存储 | 实时推送后丢弃 |

**清理逻辑**：
- 清理 `agent_traces` 时，需排除正在运行的任务
- 清理 `phase_executions` 时，需保留最近 30 天内的记录
- 使用 TTL 索引自动清理过期数据

**理由**：
- 7 天足够用户发现并报告问题
- 30 天的统计数据可用于系统优化分析
- 最终报告永久保留，满足用户查阅需求
- LLM 消息流数据量大、回看价值低，不适合持久化

## 三、WebSocket 事件设计

### 3.1 事件类型扩展

在现有事件基础上，新增以下事件：

#### 3.1.1 AGENT_MESSAGE（智能体消息）

**触发时机**：智能体与 LLM 交互时，每产生一条消息就推送

**用途**：实时展示智能体的"思考过程"，让用户了解 AI 在做什么

**数据结构**：
```
{
  "event_type": "agent_message",
  "task_id": "xxx",
  "timestamp": 1234567890,
  "data": {
    "agent_slug": "technical_analyst",
    "agent_name": "技术分析师",
    "message_type": "user" | "assistant" | "tool" | "system",
    "content": "请分析 000001 的技术面...",
    "phase": 1,
    "concurrent_group": "phase1_group1",  // 所属并发组
    "is_streaming": false  // 是否为流式输出的一部分
  }
}
```

**消息类型说明**：
- `system`：系统提示词（角色定义）
- `user`：用户输入（任务描述）
- `assistant`：助手输出（AI 分析内容）
- `tool`：工具调用/返回

**处理逻辑**：
- 前端接收到后，追加到当前智能体的消息列表
- 消息列表最多保留 100 条，超过则丢弃最旧的
- 页面刷新后，消息列表清空（因为不持久化）

#### 3.1.2 PHASE_AGENTS（阶段智能体列表）

**触发时机**：阶段开始时，推送该阶段所有智能体的信息

**用途**：让前端知道这个阶段有哪些智能体、是否并发、最大并发数

**数据结构**：
```
{
  "event_type": "phase_agents",
  "task_id": "xxx",
  "timestamp": 1234567890,
  "data": {
    "phase": 1,
    "phase_name": "分析师团队",
    "execution_mode": "concurrent",  // "concurrent" 或 "serial"
    "max_concurrency": 3,
    "concurrent_group": "phase1_group1",  // 并发组 ID
    "agents": [
      {
        "slug": "technical_analyst",
        "name": "技术分析师",
        "position": 1,  // 在并发组中的位置
        "color": "#409EFF"  // 前端用于区分不同智能体
      },
      // ...
    ]
  }
}
```

#### 3.1.3 CONCURRENT_GROUP_STARTED（并发组开始）

**触发时机**：并发组启动时推送

**用途**：明确通知前端一组智能体即将并发执行

**数据结构**：
```
{
  "event_type": "concurrent_group_started",
  "task_id": "xxx",
  "timestamp": 1234567890,
  "data": {
    "group_id": "phase1_group1",
    "phase": 1,
    "agents": ["technical_analyst", "fundamental_analyst", "sentiment_analyst"],
    "max_concurrency": 3,
    "estimated_duration_sec": 120  // 预估耗时
  }
}
```

#### 3.1.4 TOOL_CALLED 增强

**现有事件**，需要增加以下字段：
- `trace_id`：用于关联 tool_called 和 tool_result
- `phase`：所属阶段
- `concurrent_group`：所属并发组（如果有）

#### 3.1.5 AGENT_THINKING（可选，用于思考模式）

**触发时机**：当 AI 模型支持 thinking 模式时，推送思考过程

**数据结构**：
```
{
  "event_type": "agent_thinking",
  "task_id": "xxx",
  "timestamp": 1234567890,
  "data": {
    "agent_slug": "technical_analyst",
    "thinking": "我需要先获取该股票的 K 线数据...",
    "phase": 1,
    "concurrent_group": "phase1_group1"
  }
}
```

### 3.2 事件推送的时机和频率

| 事件 | 触发时机 | 推送频率限制 | 并发相关 |
|------|---------|-------------|---------|
| task_started | 任务开始 | 1 次 | - |
| phase_started | 阶段开始 | 每阶段 1 次 | - |
| phase_agents | 阶段开始（紧跟 phase_started） | 每阶段 1 次 | **包含并发信息** |
| concurrent_group_started | 并发组启动 | 每并发组 1 次 | **并发专用** |
| agent_started | 智能体开始 | 每智能体 1 次 | **并发时同时推送多个** |
| agent_message | LLM 产生消息 | 不限制（流式） | **每个智能体独立推送** |
| tool_called | 工具调用前 | 不限制 | **可能同时发生多个** |
| tool_result | 工具调用完成 | 不限制 | **可能同时发生多个** |
| agent_completed | 智能体完成 | 每智能体 1 次 | **并发时可能同时完成多个** |
| concurrent_group_completed | 并发组全部完成 | 每并发组 1 次 | **并发专用** |
| phase_completed | 阶段完成 | 每阶段 1 次 | - |
| task_completed | 任务完成 | 1 次 | - |

### 3.3 并发场景下的特殊处理

#### 3.3.1 多个智能体同时启动

当阶段 1 启动时，3 个分析师并发启动，前端会在短时间内收到：
1. `phase_started` (phase=1)
2. `phase_agents` (包含 3 个智能体信息)
3. `concurrent_group_started` (group_id=phase1_group1)
4. `agent_started` (technical_analyst)
5. `agent_started` (fundamental_analyst)
6. `agent_started` (sentiment_analyst)

**前端处理**：
- 收到 `phase_agents` 后，预先渲染 3 个智能体卡片（状态为 pending）
- 收到 `concurrent_group_started` 后，用视觉方式（如连线）将它们分组
- 收到每个 `agent_started` 后，逐个更新卡片状态为 running

#### 3.3.2 多个智能体同时完成

并发组中的智能体可能在不同时间完成：

**场景 A：依次完成**
- 技术分析师完成 → 推送 `agent_completed` → 前端更新该卡片为 completed
- 基本面分析师完成 → 推送 `agent_completed` → 前端更新该卡片为 completed
- 情绪分析师完成 → 推送 `agent_completed` → 前端更新该卡片为 completed
- 全部完成后 → 推送 `concurrent_group_completed` → 前端可以进行阶段切换动画

**场景 B：同时完成**
- 如果多个智能体几乎同时完成，前端会在短时间内收到多个 `agent_completed`
- 前端应批量处理这些事件，避免频繁重绘

## 四、前端交互设计

### 4.1 页面布局

整体采用**三段式布局**：

```
┌────────────────────────────────────────────────────────────────┐
│  顶部：任务基本信息栏                                            │
│  股票代码 | 状态 | 进度条 | 已耗时 | Token 成本 | [停止] [返回]   │
├────────────────────────────────────────────────────────────────┤
│  中部：左右分栏布局                                              │
│  ┌─────────────────────────────┬─────────────────────────────┐ │
│  │  左侧 (2/3)                 │  右侧 (1/3)                 │ │
│  │                             │                             │ │
│  │  1. 阶段时间线              │  1. 工具调用统计            │ │
│  │     - 并发智能体组（横向）  │     - 按智能体分组          │ │
│  │     - 串行智能体组（纵向）  │     - 按工具分组            │ │
│  │                             │     - 最近调用记录          │ │
│  │  2. 当前智能体详情（可折叠）│                             │ │
│  │     - LLM 消息流            │  2. Token 使用明细          │ │
│  │     - 实时工具调用          │                             │ │
│  │                             │                             │ │
│  │  3. 最终报告                │                             │ │
│  └─────────────────────────────┴─────────────────────────────┘ │
└────────────────────────────────────────────────────────────────┘
```

### 4.2 阶段时间线设计

#### 4.2.1 并发智能体组（阶段 1、3）

**视觉呈现**：
```
┌─ 阶段 1：分析师团队 ─────────────────────────┐
│ ┌─────────────────────────────────────────┐ │
│ │ 并发组 (最多 3 个同时执行)               │ │
│ │ ┌──────────┐ ┌──────────┐ ┌──────────┐  │ │
│ │ │ 技术分析师│ │基本面分析师│ │情绪分析师│  │ │
│ │ │ [运行中]  │ │ [完成]   │ │ [完成]   │  │ │
│ │ │ ▬▬▬▬▬▬▬ │ │ ▬▬▬▬▬▬▬ │ │ ▬▬▬▬▬▬▬ │  │ │
│ │ │ 2.3s     │ │ 1.8s     │ │ 2.1s     │  │ │
│ │ └──────────┘ └──────────┘ └──────────┘  │ │
│ │         ↑         ↑         ↑           │ │
│ │     同时启动，独立执行              │ │
│ └─────────────────────────────────────────┘ │
│ 完成进度: 2/3 (67%)                         │
└─────────────────────────────────────────────┘
```

**交互逻辑**：
- 点击任一智能体卡片，在下方展开该智能体的实时执行详情
- 卡片状态指示器：
  - `pending`：灰色边框，显示"等待中"
  - `running`：蓝色边框 + 脉动动画，显示进度条
  - `completed`：绿色边框，显示耗时
  - `failed`：红色边框，显示错误图标

**并发展示细节**：
- 同一并发组的智能体卡片之间有细线连接，表示"同时进行"
- 进度条是独立的，每个智能体有自己的进度
- 组底部显示整体进度（如"完成进度: 2/3"）

#### 4.2.2 串行智能体组（阶段 2、4）

**视觉呈现**：
```
┌─ 阶段 2：研究员辩论 ───────────────────────┐
│ ┌──────────────────────────────────────┐  │
│ │ 第 1 轮 (已完成)                       │  │
│ │ ┌──────────┐         ┌──────────┐    │  │
│ │ │ 看多辩手  │  ───>   │ 看空辩手  │    │  │
│ │ │ [完成]   │         │ [完成]   │    │  │
│ │ │ 1.2s     │         │ 0.8s     │    │  │
│ │ └──────────┘         └──────────┘    │  │
│ └──────────────────────────────────────┘  │
│ ┌──────────────────────────────────────┐  │
│ │ 第 2 轮 (进行中)                       │  │
│ │ ┌──────────┐         ┌──────────┐    │  │
│ │ │ 看多辩手  │  <───   │ 看空辩手  │    │  │
│ │ │ [完成]   │         │ [运行中]  │    │  │
│ │ │ 1.5s     │         │ 0.3s     │    │  │
│ │ └──────────┘         └──────────┘    │  │
│ └──────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

**交互逻辑**：
- 用箭头表示辩论的"来回"关系

### 4.5 状态同步逻辑

#### 4.5.1 页面加载时的数据获取

用户打开任务详情页时，按以下顺序加载数据：

1. **先获取任务基本信息**（API：`GET /api/trading-agents/tasks/{id}`）
   - 包含：状态、进度、最终报告、阶段执行记录、工具统计

2. **根据阶段执行记录，渲染时间线**
   - 并发智能体横向排列
   - 串行智能体纵向排列

3. **如果任务正在运行，连接 WebSocket**
   - 订阅该任务的实时事件

4. **获取最近的工具调用记录**（API：`GET /api/trading-agents/tasks/{id}/tool-traces?limit=10`）
   - 用于展示"最近调用"

#### 4.5.2 WebSocket 事件处理

| 事件 | 前端处理动作 | 并发相关 |
|------|-------------|---------|
| phase_started | 更新当前阶段，滚动时间线到该阶段 | - |
| phase_agents | 更新该阶段的智能体列表，渲染智能体卡片 | **创建并发组** |
| concurrent_group_started | 将相关智能体卡片用视觉方式分组 | **并发专用** |
| agent_started | 将该智能体卡片状态设为 `running`，开始脉动动画 | **并发时同时更新多个** |
| agent_message | 如果当前展开的是该智能体，追加消息到详情面板 | **每个智能体独立** |
| agent_completed | 将该智能体卡片状态设为 `completed`，显示耗时 | **检查并发组是否全部完成** |
| concurrent_group_completed | 触发并发组完成动画，准备进入下一阶段 | **并发专用** |
| phase_completed | 标记阶段完成，滚动到下一阶段 | - |
| task_completed | 标记任务完成，显示最终报告，断开 WebSocket | - |
| task_failed | 标记任务失败，显示错误信息 | - |

#### 4.5.3 刷新页面后的行为

用户刷新页面时：
- **已完成的任务**：从数据库加载历史数据，静态展示
- **运行中的任务**：
  - 从数据库加载已完成部分（阶段执行记录、工具统计）
  - 连接 WebSocket，接收后续的实时事件
  - **注意**：刷新前的 LLM 消息流会丢失（因为不持久化），这是设计上的权衡

## 五、后端处理逻辑

### 5.1 智能体执行时的数据推送

#### 5.1.1 智能体基类增强

在智能体基类中增加"消息推送钩子"，在 LLM 交互的关键节点推送事件：

**推送节点**：
1. 系统提示词构建完成后 → `agent_message(type=system)`
2. 用户输入发送前 → `agent_message(type=user)`
3. 助手输出开始 → `agent_message(type=assistant, 开始流式输出)`
4. 助手输出每个 chunk → `agent_message(type=assistant, 追加内容)`
5. 工具调用前 → `tool_called`
6. 工具调用完成后 → `tool_result`
7. 助手输出结束 → `agent_completed`

#### 5.1.2 消息截断策略

为避免消息过大导致网络/性能问题，需要对消息进行截断：

| 消息类型 | 最大长度 | 截断方式 |
|---------|---------|---------|
| system (系统提示词) | 5000 字符 | 直接截断，末尾加 `...` |
| user (用户输入) | 2000 字符 | 直接截断 |
| assistant (助手输出) | 不限制 | 实时流式推送，前端自行处理 |
| tool_input | 5000 字符 | JSON 序列化后截断 |
| tool_output | 10000 字符 | JSON 序列化后截断 |

**注意**：截断仅影响 WebSocket 推送，数据库中存储完整数据（如果需要存储）。

### 5.2 工具调用追踪逻辑

#### 5.2.1 追踪流程

```
┌─────────────────────────────────────────────────────────────┐
│  智能体调用工具                                              │
│     │                                                        │
│     ├─> 1. 推送 tool_called 事件 (WebSocket)                 │
│     │                                                        │
│     ├─> 2. 记录开始时间                                      │
│     │                                                        │
│     ├─> 3. 执行工具函数                                     │
│     │     │                                                  │
│     │     ├─> 成功: 返回结果                                │
│     │     └─> 失败: 记录错误                                │
│     │                                                        │
│     ├─> 4. 推送 tool_result 事件 (WebSocket)                 │
│     │                                                        │
│     ├─> 5. 异步写入 agent_traces 集合 (不阻塞主流程)         │
│     │                                                        │
│     └─> 6. 异步更新 task_doc.tool_stats (不阻塞主流程)       │
└─────────────────────────────────────────────────────────────┘
```

#### 5.2.2 并发场景下的追踪

当多个智能体并发执行时，工具调用可能**同时发生**：

**场景**：技术分析师调用 get_kline_data，基本面分析师同时调用 get_company_info

**追踪要点**：
1. 每个工具调用都有独立的 `trace_id`
2. `concurrent_group` 字段记录所属并发组
3. 写入数据库时使用异步任务，避免阻塞
4. 统计更新使用 MongoDB 的原子操作，避免并发问题

**数据示例**：
```javascript
// agent_traces 集合中的两条记录（时间戳几乎相同）
{
  "trace_id": "aaa-111",
  "agent_slug": "technical_analyst",
  "tool_name": "get_kline_data",
  "started_at": "2024-01-01T10:00:10.000Z",
  "concurrent_group": "phase1_group1"
},
{
  "trace_id": "bbb-222",
  "agent_slug": "fundamental_analyst",
  "tool_name": "get_company_info",
  "started_at": "2024-01-01T10:00:10.050Z",  // 50ms 后
  "concurrent_group": "phase1_group1"
}
```

#### 5.2.3 统计更新逻辑

每次工具调用完成后，更新 `tool_stats`：

```python
# 伪代码
async def update_tool_stats(task_id, agent_slug, tool_name, success, duration_ms):
    # 使用 MongoDB 的原子操作更新，避免并发问题
    await db.analysis_tasks.update_one(
        {"_id": ObjectId(task_id)},
        {
            "$inc": {
                f"tool_stats.{agent_slug}.total_calls": 1,
                f"tool_stats.{agent_slug}.total_duration_ms": duration_ms,
                f"tool_stats.{agent_slug}.tools.{tool_name}.count": 1,
                f"tool_stats.{agent_slug}.tools.{tool_name}.total_duration_ms": duration_ms,
                f"tool_stats.{agent_slug}.tools.{tool_name}.success_count" if success else "fail_count": 1,
            }
        }
    )
```

**并发安全**：使用 `$inc` 操作符，多个并发更新不会互相覆盖

### 5.3 阶段执行记录的维护

#### 5.3.1 记录创建时机

| 时机 | 动作 | 数据库操作 | 并发相关 |
|------|------|-----------|---------|
| 阶段开始 | 在 `phase_executions` 中创建新记录 | `$push` | 设置 `execution_mode` 和 `max_concurrency` |
| 智能体开始 | 找到对应阶段和智能体，更新状态 | `$set` | 并发时同时更新多个智能体 |
| 智能体完成 | 更新智能体的 `completed_at` 和 `token_usage` | `$set` | 并发时可能同时完成多个 |
| 阶段完成 | 更新阶段的 `completed_at` | `$set` | 检查所有智能体是否完成 |

#### 5.3.2 并发组信息生成

在任务启动时，根据智能体配置预先生成 `concurrency_groups`：

- **阶段 1**：根据用户选择的智能体，生成一个并发组（最多 3 个）
- **阶段 2**：看多/看空辩手，属于串行执行，不生成并发组
- **阶段 3**：激进/保守/中性三方，属于并发组（3 个）
- **阶段 4**：总结智能体，单独执行

**生成逻辑**：
```
for phase in [1, 2, 3, 4]:
    if phase == 1:
        # 用户选择的智能体并发执行
        group_id = f"phase{phase}_group1"
        agents = user_selected_agents  # 如 ["technical", "fundamental", "sentiment"]
        create_concurrent_group(group_id, phase, agents, max_concurrency=3)

    elif phase == 2:
        # 串行执行，不创建并发组
        pass

    elif phase == 3:
        # 三方并发
        group_id = f"phase{phase}_group1"
        agents = ["aggressive", "conservative", "neutral"]
        create_concurrent_group(group_id, phase, agents, max_concurrency=3)

    elif phase == 4:
        # 单独执行
        pass
```

### 5.4 错误处理和降级

#### 5.4.1 WebSocket 推送失败

如果 WebSocket 推送失败（如客户端断开）：
- 记录错误日志
- **不影响任务执行**，推送是"尽力而为"
- 客户端重连后，通过轮询 API 获取最新状态

#### 5.4.2 工具调用追踪写入失败

如果 `agent_traces` 写入失败：
- 记录错误日志
- **不影响任务执行**
- 统计数据仍然会更新（使用原子操作）

#### 5.4.3 并发场景下的错误处理

**场景**：并发组中某个智能体失败

**处理逻辑**：
1. 标记该智能体状态为 `failed`
2. 记录错误信息到 `phase_executions`
3. **不影响其他并发智能体**，它们继续执行
4. 当所有智能体都完成（包括失败的）后，推送 `concurrent_group_completed`
5. 根据失败策略决定是否继续下一阶段

### 5.5 数据清理任务

#### 5.5.1 清理策略

| 数据 | 保留时间 | 清理频率 | 清理方式 |
|------|---------|---------|---------|
| agent_traces | 7 天 | 每天凌晨 3 点 | 删除 `timestamp < 7天前` 的记录 |
| phase_executions | 30 天 | 每天凌晨 3 点 | 从任务文档中删除该字段 |
| tool_stats | 永久 | 不清理 | 不清理 |

**注意**：清理 `agent_traces` 时，不能删除正在运行的任务的追踪记录。

#### 5.5.2 清理逻辑

```
1. 查询所有 running 状态的任务 ID 列表
2. 删除 agent_traces 中：
   - timestamp < 7 天前
   - AND task_id NOT IN (running 任务列表)
3. 更新已完成任务的阶段执行记录：
   - 如果 completed_at < 30 天前
   - 将 phase_executions 设为空数组
```

## 六、并发展示的特殊处理

### 6.1 并发与串行的判断依据

**并发模式**（阶段 1、3）：
- 多个智能体**同时启动**（`agent_started` 事件在短时间内连续触发）
- 前端检测：如果 3 秒内收到多个 `agent_started` 事件，视为并发
- 后端明确告知：`phase_agents` 事件中包含 `execution_mode: "concurrent"`

**串行模式**（阶段 2、4）：
- 智能体**依次启动**（下一个 `agent_started` 在上一个 `agent_completed` 之后）
- 前端检测：严格按照完成-开始的顺序
- 后端明确告知：`phase_agents` 事件中包含 `execution_mode: "serial"`

### 6.2 并发进度计算

当多个智能体并发执行时，如何计算整体进度？

**方案**：
```
整体进度 = Σ(各阶段进度)

阶段进度 = (已完成智能体数量 / 总智能体数量) × 该阶段权重

例如：阶段 1 有 3 个智能体，阶段权重 25%
- 0 个完成：0%
- 1 个完成：8.33% (1/3 × 25%)
- 2 个完成：16.67% (2/3 × 25%)
- 3 个完成：25% (3/3 × 25%)

假设各阶段权重：
- 阶段 1：25%（分析师团队）
- 阶段 2：35%（研究员辩论）
- 阶段 3：25%（风险评估）
- 阶段 4：15%（总结输出）

如果阶段 1 完成 2/3，阶段 2 未开始：
整体进度 = 16.67% + 0 + 0 + 0 = 16.67%
```

### 6.3 并发智能体的状态展示

当多个智能体同时处于 `running` 状态时：

1. **默认展开**：第一个进入 `running` 状态的智能体
2. **切换逻辑**：用户点击另一个智能体卡片时，切换到该智能体的详情
3. **自动切换**：当前智能体完成后，自动展开下一个仍在运行的智能体
4. **并发提示**：面板顶部显示其他运行中的智能体列表

### 6.4 辩论轮次的展示

阶段 2 的辩论有**多轮**，如何展示？

**方案**：
```
┌─ 阶段 2：研究员辩论 ─────────────────────────┐
│  第 1 轮 (3/3 完成)                           │
│  ┌──────────┐    ┌──────────┐               │
│  │ 看多     │ -> │ 看空     │               │
│  │ ✅       │    │ ✅       │               │
│  └──────────┘    └──────────┘               │
│                                             │
│  第 2 轮 (2/3 完成)                           │
│  ┌──────────┐    ┌──────────┐               │
│  │ 看多     │ <- │ 看空     │               │
│  │ ✅       │    │ 🔄 运行中 │               │
│  └──────────┘    └──────────┘               │
│                                             │
│  第 3 轮 (等待中)                             │
│  ┌──────────┐    ┌──────────┐               │
│  │ 看多     │ -> │ 看空     │               │
│  │ ⏳       │    │ ⏳       │               │
│  └──────────┘    └──────────┘               │
└─────────────────────────────────────────────┘
```

- 每轮显示一个独立的区域
- 用箭头方向表示"反驳"关系
- 轮次标题显示进度（如 `3/3 完成`）

### 6.5 并发组完成检测

**问题**：如何判断并发组的所有智能体都已完成？

**方案**：
1. 后端在收到每个 `agent_completed` 时，检查该并发组是否还有 `running` 状态的智能体
2. 如果全部完成，推送 `concurrent_group_completed` 事件
3. 前端收到后，可以进行完成动画，并准备进入下一阶段

**伪代码**：
```python
async def on_agent_completed(task_id, agent_slug, concurrent_group):
    # 更新智能体状态
    await update_agent_status(task_id, agent_slug, "completed")

    # 检查并发组是否全部完成
    if concurrent_group:
        running_agents = await get_running_agents_in_group(task_id, concurrent_group)
        if len(running_agents) == 0:
            # 推送并发组完成事件
            await broadcast_event(concurrent_group_completed(
                task_id=task_id,
                group_id=concurrent_group
            ))
```

## 七、性能考虑

### 7.1 数据库写入优化

| 写入操作 | 优化方案 | 并发相关 |
|---------|---------|---------|
| agent_traces 插入 | 异步执行，不阻塞主流程 | 并发时可能有多个同时插入 |
| tool_stats 更新 | 使用 `$inc` 原子操作，避免读取-修改-写入 | 并发时多个智能体同时更新 |
| phase_executions 更新 | 批量更新，减少数据库操作次数 | 并发时同时更新多个智能体状态 |

### 7.2 WebSocket 推送优化

| 优化点 | 方案 | 并发相关 |
|-------|------|---------|
| 消息大小 | 截断超长内容（如 tool_output 限制 10000 字符） | - |
| 推送频率 | `agent_message` 可配置开关，用户关闭时不推送 | - |
| 批量推送 | 多个消息合并为一个 JSON 数组推送 | 并发完成时可能有多条 `agent_completed` |
| 事件节流 | 相同事件在短时间内只推送一次（如进度更新） | - |

### 7.3 前端渲染优化

| 优化点 | 方案 | 并发相关 |
|-------|------|---------|
| 消息列表 | 虚拟滚动，只渲染可见区域的消息 | 多个智能体同时推送消息时 |
| 工具调用记录 | 分页加载，每页 20 条 | - |
| 大量 DOM | 使用 `v-show` 而非 `v-if`，避免频繁创建/销毁 | 并发组的多个智能体卡片 |
| 动画性能 | 使用 CSS transform 而非 top/left，利用 GPU 加速 | 并发完成时的动画 |

### 7.4 并发场景下的特殊优化

**场景**：3 个智能体同时推送消息

**问题**：如果每个智能体每秒推送 10 条消息，总共 30 条/秒，可能造成性能问题

**优化方案**：
1. **消息缓冲**：将短时间内的多条消息合并为一批推送
2. **优先级队列**：重要消息（如 agent_completed）优先推送
3. **用户配置**：允许用户关闭"实时消息流"，只看关键事件

- **屏幕阅读器**：为智能体卡片添加 ARIA 标签

### 8.4 并发场景下的用户体验

**场景 1：用户想同时查看多个并发智能体的状态**

## 九、扩展性考虑

### 9.1 未来可扩展的功能

1. **导出执行报告**：将整个执行过程导出为 PDF/HTML
2. **对比分析**：选择两个任务，对比它们的执行差异
3. **工具调用性能分析**：识别响应慢的工具，优化 MCP 服务器
4. **智能体效果评估**：统计哪些智能体最常产生正面/负面推荐
5. **并发热力图**：可视化展示并发执行的时间分布

### 9.2 数据模型扩展性

- `phase_executions` 使用数组结构，便于添加新字段（如 `error_count`、`retry_count`）
- `tool_stats` 的嵌套结构支持按任意维度聚合
- `agent_traces` 独立存储，便于后续分析和清理
- `concurrency_groups` 可以支持更复杂的并发模式（如分层并发）

### 9.3 支持更复杂的并发模式

当前设计支持：
- **简单并发**：多个智能体同时启动、独立执行
- **串行**：智能体依次执行
- **混合**：并发组和串行组交替

未来可支持：
- **分层并发**：组内并发，组间串行
- **依赖并发**：智能体 A 完成后，B 和 C 才能并发启动
- **动态并发**：根据执行结果动态调整并发数
