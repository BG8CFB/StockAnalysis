## 一、并发控制概述

### 1.1 设计目标

| 目标 | 描述 |
|------|------|
| **资源保护** | 防止模型服务器过载 |
| **公平调度** | 多用户任务公平分配资源 |
| **可配置性** | 通过配置灵活调整并发参数 |
| **可视化** | 用户可查看队列位置和等待时间 |

### 1.2 核心组件

| 组件 | 文件位置 | 核心职责 |
|------|----------|----------|
| **ConcurrencyController** | `manager/concurrency_controller.py` | 模型级并发控制、任务排队 |
| **BatchTaskManager** | `manager/batch_manager.py` | 批量任务并发控制、任务调度 |

## 二、三层并发架构

### 2.1 架构图

```
┌─────────────────────────────────────────────────────────────────────────┐
│                         TradingAgents 并发控制                          │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                           │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │ 第一层：模型级并发 (max_concurrency)                                │ │
│  │ 限制对单个 AI 模型的最大并发请求数                                   │ │
│  │ 例如：max_concurrency=40                                            │ │
│  └────────────────────────────────────────────────────────────────────┘ │
│                            ↓                                           │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │ 第二层：任务级并发 (task_concurrency)                               │ │
│  │ 定义单个任务消耗的并发槽位数                                          │ │
│  │ 可同时运行的任务数 = max_concurrency / task_concurrency              │ │
│  │ 例如：task_concurrency=2 → 可运行 20 个任务                          │ │
│  └────────────────────────────────────────────────────────────────────┘ │
│                            ↓                                           │
│  ┌────────────────────────────────────────────────────────────────────┐ │
│  │ 第三层：批量任务并发 (batch_concurrency)                            │ │
│  │ 限制批量任务中同时执行的子任务数                                     │ │
│  │ 例如：batch_concurrency=5 → 批量任务最多运行 5 个子任务              │ │
│  └────────────────────────────────────────────────────────────────────┘ │
│                                                                           │
└─────────────────────────────────────────────────────────────────────────┘
```

### 2.2 并发参数配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_concurrency` | 40 | 模型最大并发连接数 |
| `task_concurrency` | 2 | 单任务消耗的槽位数 |
| `batch_concurrency` | 1 | 批量任务同时运行的子任务数 |

**可同时运行的任务数计算**：
```
max_running_tasks = max_concurrency / task_concurrency
例如：40 / 2 = 20 个任务可同时运行
```

## 三、模型级并发控制

### 3.1 工作原理

```
用户提交任务
      ↓
请求执行槽位 (request_execution)
      ↓
┌─────────────────────────────────────────┐
│ 检查当前活跃任务数                      │
│ active_count < max_running_tasks?       │
└─────────────────────────────────────────┘
      ↓
   是 → 执行任务，标记为活跃
   否 → 加入等待队列（FIFO）
```

### 3.2 Redis 数据结构

| 数据结构 | 类型 | 说明 |
|----------|------|------|
| `concurrency:model:{model_id}:active` | Hash | 活跃任务：{task_id: user_id} |
| `concurrency:model:{model_id}:waiting` | List | 等待队列（FIFO） |
| `concurrency:wakeup:{model_id}` | Pub/Sub | 唤醒通道（事件通知） |

### 3.3 执行请求流程

```python
async def request_execution(
    model_id: str,
    task_id: str,
    user_id: str,
    model_config: Dict[str, Any],
) -> Dict[str, Any]:
    """
    请求执行任务

    返回:
    {
        "can_execute": bool,      # 是否可以立即执行
        "queue_position": int,    # 队列位置（0=可执行）
        "waiting_count": int,     # 总等待任务数
        "reason": str,            # 不能执行的原因
    }
    """
```

### 3.4 等待执行机制

**轮询机制**（当前实现）：
- 每 2 秒检查一次槽位是否可用
- 超时时间：300 秒（5 分钟）
- 超时后抛出 `asyncio.TimeoutError`

```python
async def wait_for_execution(
    model_id: str,
    task_id: str,
    user_id: str,
    model_config: Dict[str, Any],
    timeout: float = 300.0,
) -> Dict[str, Any]:
    """等待任务可以执行"""
```

### 3.5 资源释放

```python
async def release_execution(
    model_id: str,
    task_id: str,
    user_id: str,
    batch_id: Optional[str] = None,
):
    """释放任务执行资源"""
```

## 四、任务级并发控制

### 4.1 任务并发定义

**task_concurrency** 定义单个任务消耗的并发槽位数：

| task_concurrency 值 | 含义 | 可同时运行任务数（max=40） |
|---------------------|------|---------------------------|
| 1 | 轻量级任务 | 40 |
| 2 | 标准任务（默认） | 20 |
| 4 | 重量级任务 | 10 |

### 4.2 任务槽位计算

```
max_running_tasks = max_concurrency // task_concurrency

示例：
max_concurrency = 40
task_concurrency = 2
max_running_tasks = 40 // 2 = 20 个任务
```

### 4.3 活跃任务追踪

使用 Redis Hash 存储活跃任务：
```
Key: concurrency:model:{model_id}:active
Field: task_id
Value: user_id
```

## 五、批量任务并发控制

### 5.1 批量任务定义

用户可以一次性提交多只股票的分析请求（1-50 只）。

### 5.2 批量并发限制

**batch_concurrency** 限制批量任务中同时执行的子任务数：

| batch_concurrency 值 | 含义 | 同时运行的子任务数 |
|----------------------|------|-------------------|
| 1 | 串行执行（默认） | 1 |
| 5 | 标准并发 | 5 |
| 10 | 高并发 | 10 |

### 5.3 批量任务执行流程

```
用户提交批量任务（10 只股票）
      ↓
创建 BatchTaskContext
      ↓
第一批：创建 min(batch_concurrency, 10) 个任务
      ↓
┌─────────────────────────────────────────┐
│ 任务完成回调 (on_task_completed)        │
│ ↓                                        │
│ 检查是否有待处理股票                     │
│ ↓                                        │
│ 创建下一批任务                           │
│ ↓                                        │
│ 重复直到所有任务完成                     │
└─────────────────────────────────────────┘
```

### 5.4 批量任务状态

```python
@dataclass
class BatchTaskContext:
    batch_id: str
    user_id: str
    stock_codes: List[str]           # 所有待处理的股票代码
    created_tasks: List[str]          # 已创建的任务 ID
    pending_stocks: List[str]         # 待处理的股票代码
    running_count: int                # 当前运行中的任务数
    max_concurrent: int               # 最大并发数
```

### 5.5 用户批量任务隔离

使用 Redis List 追踪用户的活跃批量任务：
```
Key: concurrency:user:{user_id}:batch_active
Value: [batch_id_1, batch_id_2, ...]
```

## 六、单任务内并发

### 6.1 Phase 1 并发控制

Phase 1 分析师团队的并发由智能体配置控制：

```yaml
# phases/phase1/agents/...
phase1:
  enabled: true
  max_concurrency: 3  # 同时运行的分析师数量
```

### 6.2 单任务并发示例

**场景**：用户配置了 6 个分析师，但 `max_concurrency=3`

```
任务启动，Phase 1 有 6 个分析师
      ↓
┌─────────────────────────────────────────┐
│ 并发控制（max_concurrency=3）            │
│ ↓                                        │
│ 批次 1：分析师 1、2、3 并行执行          │
│ 批次 2：分析师 4、5、6 并行执行          │
└─────────────────────────────────────────┘
```

### 6.3 并发配置说明

| 参数 | 位置 | 默认值 | 说明 |
|------|------|--------|------|
| `max_concurrency` | phase1 配置 | 3 | Phase 1 同时运行的智能体数 |
| `model_concurrency` | 模型配置 | 40 | 模型级最大并发数 |

## 七、队列管理机制

### 7.1 FIFO 队列

当并发限制达到上限时，新任务进入 FIFO（先进先出）队列等待：

```
Redis List: concurrency:model:{model_id}:waiting
操作：
  - 入队：rpush (队尾添加)
  - 出队：lpop (队头移除)
  - 查询：lrange (查看所有)
```

### 7.2 队列位置查询

用户可以查询自己在队列中的位置：

```python
async def get_queue_position(
    model_id: str,
    task_id: str,
) -> Dict[str, int]:
    """
    获取任务在队列中的位置

    返回:
    {
        "position": int,      # 队列位置（0=可执行，>0=前面有任务）
        "waiting_count": int, # 总等待任务数
    }
    """
```

### 7.3 队列行为

| 场景 | 行为 |
|------|------|
| 新任务提交，槽位可用 | 立即执行，`queue_position=0` |
| 新任务提交，槽位满 | 加入队列，`queue_position=N` |
| 任务完成 | 从活跃任务移除，队列头任务被唤醒 |
| 任务取消 | 从队列中移除，不影响其他任务 |

### 7.4 多用户场景

**单用户提交多个任务**：
```
用户 A 提交 120 个任务
      ↓
┌─────────────────────────────────────────┐
│ 模型并发限制: max_concurrency=40        │
│ 任务并发: task_concurrency=2            │
│ 可运行任务数: 20                        │
└─────────────────────────────────────────┘
      ↓
┌────── 20 个任务立即执行 ──────┐
└────────────────────────────────┘
      ↓
┌────── 100 个任务进入队列 ─────┐
└────────────────────────────────┘
      ↓
任务完成 → 释放槽位 → 队列头任务执行
```

**多用户提交任务**：
```
用户 A: 20 个任务
用户 B: 15 个任务
用户 C: 30 个任务
      ↓
所有用户共享模型级并发槽位
      ↓
按提交时间顺序执行（FIFO）
```

## 八、并发场景示例

### 8.1 单用户多任务并发

```
用户提交 50 个单股分析任务
      ↓
并发限制：max_concurrency=40, task_concurrency=2
可运行任务数 = 20
      ↓
┌────── 20 个任务立即执行 ──────┐
└────────────────────────────────┘
┌────── 30 个任务进入队列 ──────┐
└────────────────────────────────┘
      ↓
任务完成 → 队列任务依次执行
```

### 8.2 批量任务并发

```
用户提交批量任务（分析 10 只股票）
      ↓
并发限制：batch_concurrency=5
      ↓
┌────── 第一批：5 个任务 ──────┐
└────────────────────────────────┘
      ↓
单个任务完成 → 触发下一批
      ↓
┌────── 第二批：1 个任务 ──────┐（剩余 4 只股票的某个完成后）
└────────────────────────────────┘
```

### 8.3 单任务内并发（Phase 1）

```
任务启动，Phase 1 有 6 个分析师
      ↓
并发限制：phase1.max_concurrency=3
      ↓
┌──── 批次 1：分析师 1、2、3 ────┐
└────────────────────────────────┘
      ↓
┌──── 批次 2：分析师 4、5、6 ────┐
└────────────────────────────────┘
```
