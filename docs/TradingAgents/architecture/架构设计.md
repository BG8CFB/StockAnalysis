## 一、架构概述

### 1.1 核心理念

TradingAgents 模块模拟专业投资分析团队的工作模式，通过结构化的多阶段分析流程，对投资标的进行深度研究，最终形成兼具深度、广度和风险考量的投资决策建议。

**设计原则**：
- **简单函数式调度**：使用 Python 函数直接调用各阶段
- **LangChain create_agent**：使用标准接口创建智能体
- **清晰可读**：调度文件一目了然，易于维护和调试
- **高效执行**：利用 asyncio 实现真正的并行处理
- **组件解耦**：每个阶段独立运行，易于测试和扩展

**核心工作模式**：
- **并行研究**：多个不同领域的分析师同时从各自的专业角度进行初步分析
- **交叉辩论**：设立正反方角色，通过多轮辩论深挖潜在机会和风险
- **多层决策**：设置"经理"角色，对下级辩论进行总结和裁决
- **风险前置**：在形成最终决策前，进行独立的多视角风险评估
- **统一输出**：汇总所有过程信息，生成结构化的最终报告

## 三、核心组件

### 3.1 任务管理器 (TaskManager)

**职责**：
- 创建和初始化任务
- 管理任务生命周期（pending → running → completed/failed）
- 处理任务取消和停止
- 任务状态查询和更新
- 批量任务管理

**核心方法**：

| 方法 | 描述 |
|------|------|
| `create_task()` | 创建单个分析任务 |
| `create_batch_task()` | 创建批量分析任务 |
| `get_task_status()` | 获取任务状态 |
| `list_tasks()` | 列出用户任务 |
| `cancel_task()` | 取消任务 |
| `stop_task()` | 停止运行中的任务 |
| `execute_analysis_workflow()` | 执行分析工作流（调用 scheduler） |
| `complete_task()` | 标记任务完成 |
| `fail_task()` | 标记任务失败 |

**批量任务失败处理**：

批量任务中单个子任务失败时，系统自动执行以下处理：

- **自动重试**：失败任务自动重试，最多重试 3 次
- **重试间隔**：第1次立即重试，第2次等待 5 秒，第3次等待 15 秒
- **失败标记**：重试 3 次仍失败后，标记该子任务状态为 `failed`
- **其他继续**：单个子任务失败不影响其他子任务执行
- **部分完成**：批量任务可以部分成功，用户可查看成功和失败的任务列表
- **失败汇总**：批量任务完成后，汇总失败原因和任务列表

### 3.2 并发控制器 (ConcurrencyController)

**职责**：
- 管理模型级别的并发控制
- 实现任务排队机制（FIFO）
- 处理并发槽位的申请和释放
- 支持批量任务并发限制

**三层并发架构**：

```
┌────────────────────────────────────────────────────────────┐
│                    并发控制层级                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 第一层：模型级并发 (max_concurrency)                  │ │
│  │ 限制对单个 AI 模型的最大并发请求数                    │ │
│  │ 例如：max_concurrency=40                              │ │
│  └──────────────────────────────────────────────────────┘ │
│                            ↓                               │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 第二层：任务级并发 (task_concurrency)                 │ │
│  │ 定义单个任务消耗的并发槽位数                           │ │
│  │ 可同时运行的任务数 = max_concurrency / task_concurrency│ │
│  │ 例如：task_concurrency=2 → 可运行 20 个任务           │ │
│  └──────────────────────────────────────────────────────┘ │
│                            ↓                               │
│  ┌──────────────────────────────────────────────────────┐ │
│  │ 第三层：批量任务并发 (batch_concurrency)               │ │
│  │ 限制批量任务中同时执行的子任务数                       │ │
│  │ 例如：batch_concurrency=5 → 批量任务最多运行 5 个子任务│ │
│  └──────────────────────────────────────────────────────┘ │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 3.3 工作流调度器 (WorkflowScheduler)

**职责**：
- 执行四阶段流程（Python 原生 async/await）
- 管理状态转换和累积
- 处理阶段启用/禁用逻辑
- 协调并行和串行执行

**执行流程**：

1. **初始化**：创建分析状态对象，初始化模型
2. **Phase 1**：并行执行 6 个分析师，收集分析报告
3. **Phase 2**：串行执行观点辩论，看涨和看跌分析师交替发言
4. **Phase 3**：专业交易员制定交易执行计划
5. **Phase 4**：并行执行策略分析和风险评估
6. **完成**：标记任务状态为 completed，返回最终结果

**状态管理**：

| 字段 | 类型 | 累积模式 | 描述 |
|------|------|----------|------|
| `user_id` | str | 覆盖 | 用户 ID |
| `task_id` | str | 覆盖 | 任务 ID |
| `stock_code` | str | 覆盖 | 股票代码 |
| `analyst_reports` | list | 累积 | Phase 1 分析报告 |
| `debate_turns` | list | 累积 | Phase 2 辩论记录 |
| `trading_plan` | object | 覆盖 | Phase 3 交易计划 |
| `strategy_reports` | list | 累积 | Phase 4 策略报告 |
| `final_report` | str | 覆盖 | 最终报告 |
| `token_usage` | list | 累积 | Token 消耗记录 |
| `tool_calls` | list | 累积 | 工具调用记录 |
| `errors` | list | 累积 | 错误记录 |

### 3.4 阶段智能体 (Phase Agents)

**Phase 1: 信息收集与基础分析（并行）**

包含 6 个分析师智能体，通过 `create_agent()` 创建：
- 财经新闻分析师：追踪市场动态，分析新闻影响
- 社交媒体分析师：挖掘社交媒体情绪，量化市场情绪
- 中国市场分析师：分析 A 股和港股市场，结合政策面分析
- 市场技术分析师：通过图表和指标预测价格走势
- 基本面分析师：深度剖析财务报表，评估内在价值
- 短线资金分析师：分析短线资金流向，识别主力意图

**Phase 2: 多空博弈与投资决策（串行）**

包含 3 个智能体，串行执行：
- 看涨分析师：构建做多投资组合方案
- 看跌分析师：寻找做空理由和卖出信号
- 投资组合经理：综合分析，做出最终决策

**Phase 3: 交易执行策划（单智能体）**

- 专业交易员：制定具体的交易执行计划，包括挂单策略和应急预案

**Phase 4: 策略风格与风险评估（并行）**

包含 4 个智能体，并行执行：
- 激进策略分析师：追求高收益，承担高波动
- 中性策略分析师：平衡风险收益，优化夏普比率
- 保守策略分析师：优先保本，防御性资产配置
- 风险管理委员会主席：审查风险，行使一票否决权

### 3.5 WebSocket 推送管理器

**职责**：
- 管理 WebSocket 连接
- 推送实时进度事件
- 处理连接认证和权限
- 连接池管理（最多 5 个/用户）

**断线重连机制**：
- **心跳检测**：每 30 秒发送一次心跳包
- **自动重连**：连接断开后自动重连（指数退避策略）
- **状态恢复**：重连成功后自动同步任务最新状态
- **最大重试**：最多重连 10 次，超过后提示用户手动刷新

## 五、技术选型

### 5.1 工作流实现

**Python 原生 async/await + LangChain create_agent**

选择理由：
- **简单直接**：无需学习复杂的编排框架
- **易于调试**：Python 原生代码，断点调试友好
- **灵活可控**：完全掌控执行流程
- **高效并行**：asyncio.gather() 实现真正的并行处理

### 5.2 LangChain 组件使用

| 组件 | 用途 | 说明 |
|------|------|------|
| `create_agent()` | 智能体创建 | LangChain 1.0 标准接口 |
| `ChatModel` | LLM 调用 | 直接调用 `.invoke()` 或 `.ainvoke()` |
| `系统提示词` | 提示词管理 | 通过 YAML 配置的 roleDefinition |
| `.bind_tools()` | 工具绑定 | 将 MCP 工具绑定到模型 |
