## 一、目录规划原则

### 1.1 分层架构（v2.0 精简版）

```
┌─────────────────────────────────────────────────────────────┐
│                    前端 (Vue 3)                            │
├─────────────────────────────────────────────────────────────┤
│                    后端 API 层                              │
│                 (HTTP + WebSocket 路由)                     │
├─────────────────────────────────────────────────────────────┤
│                    管理层 (Manager)                         │
│              (任务管理 + 业务服务 + 基础设施)                │
├─────────────────────────────────────────────────────────────┤
│                    阶段层 (Phases)                          │
│              (工作流调度 + 四阶段执行 + 实时推送)            │
├─────────────────────────────────────────────────────────────┤
│                    工具层 (Tools)                           │
│                    (MCP 工具集成)                           │
├─────────────────────────────────────────────────────────────┤
│                    配置层 (Config) + 数据层 (Models)        │
└─────────────────────────────────────────────────────────────┘
```

**架构说明**：

| 层级 | 职责 | 关键优化 |
|------|------|----------|
| **API 层** | 对外接口（HTTP + WebSocket） | 合并 websocket/ 目录 |
| **管理层** | 任务管理、业务服务、基础设施 | 合并 services/、infra/ |
| **阶段层** | 工作流调度、阶段执行、实时推送 | 合并 scheduler/、pusher/ |
| **工具层** | MCP 工具集成与过滤 | 保持独立 |
| **配置/数据层** | 配置管理、数据模型 | 保持独立 |

### 1.2 组织原则

1. **按功能模块划分**: 相关代码放在一起
2. **高内聚低耦合**: 层级清晰，职责单一
3. **前后端分离**: 独立的目录结构
4. **配置集中管理**: 统一的配置目录
5. **精简层级**: 避免过度分层，合并职责重叠的目录

### 1.3 v2.0 优化说明

**优化策略**：通过合并职责重叠的目录，将原有 10+ 个目录精简为 6 个核心目录。

| 优化项 | 原目录 | 合并至 | 理由 |
|--------|--------|--------|------|
| 服务层合并 | `services/` | `manager/` | 职责重叠，都是业务逻辑 |
| 调度层合并 | `scheduler/` | `phases/` | 调度器是阶段层的编排入口 |
| 推送层合并 | `pusher/` | `phases/` | 推送是阶段执行的一部分 |
| WebSocket 合并 | `websocket/` | `api/` | 单文件目录，属于接口层 |
| 基础设施合并 | `infra/` | `manager/` | 数据库和告警属于管理职责 |

## 二、后端目录结构（v2.0 精简版）

### 2.1 完整目录树

```
backend/modules/trading_agents/
├── __init__.py
├── admin_api.py              # 管理员 API 路由
├── schemas.py                # Pydantic 数据模型
├── exceptions.py             # 自定义异常
│
├── api/                      # API 层 - 对外接口 + 前后端连接
│   ├── __init__.py
│   ├── tasks.py              # 任务管理 API
│   ├── reports.py            # 报告查询 API
│   ├── websocket.py          # WebSocket 路由
│   └── websocket_manager.py  # WebSocket 管理器（前后端连接）
│
├── manager/                  # 管理层 - 任务管理 + 业务服务 + 基础设施
│   ├── __init__.py
│   ├── task_manager.py       # 任务管理器
│   ├── concurrency_controller.py  # 并发控制器
│   ├── batch_manager.py      # 批量任务管理器
│   ├── task_queue.py         # 任务队列
│   ├── task_expiry.py        # 任务过期处理
│   ├── task_manager_restore.py  # 任务恢复
│   ├── report_service.py     # 报告服务
│   ├── agent_config_service.py  # 智能体配置服务
│   ├── settings_service.py   # 用户设置服务
│   ├── database.py           # 数据库操作
│   └── alerts.py             # 告警处理
│
├── workflow/                 # 工作流 + 四阶段智能体（整体）
│   ├── __init__.py
│   ├── scheduler.py          # 调度文件（调度各阶段智能体）
│   ├── state.py              # 状态管理
│   ├── phase1/               # 第一阶段智能体（动态创建，可调用 tools/）
│   │   ├── __init__.py
│   │   ├── template.py       # 智能体通用模板
│   │   └── factory.py        # 智能体工厂
│   ├── phase2/               # 第二阶段智能体（固定智能体）
│   │   ├── __init__.py
│   │   ├── bull_researcher.py    # 看涨研究员
│   │   ├── bear_researcher.py    # 看跌研究员
│   │   └── research_manager.py   # 研究经理
│   ├── phase3/               # 第三阶段智能体（固定智能体）
│   │   ├── __init__.py
│   │   └── trader.py             # 交易员
│   └── phase4/               # 第四阶段智能体（固定智能体，未来可调用 tools/）
│       ├── __init__.py
│       ├── aggressive_debator.py   # 激进辩手
│       ├── neutral_debator.py     # 中立辩手
│       ├── conservative_debator.py # 保守辩手
│       ├── risk_manager.py        # 风险经理
│       └── strategies.py          # 策略
│
├── tools/                    # 工具模块
│   ├── __init__.py
│   ├── mcp/                  # MCP 连接（连接核心 MCP）
│   │   ├── __init__.py
│   │   ├── connector.py       # MCP 连接器
│   │   └── tool_filter.py     # 工具过滤器
│   └── (预留未来工具扩展)
│
├── models/                   # 数据模型 - 工作流状态
│   ├── __init__.py
│   └── state.py              # 工作流状态模型
│
└── config/                   # 配置管理 - 智能体配置
    ├── __init__.py
    ├── agents/               # 智能体配置文件
    │   ├── agents_default.yaml     # 默认配置（项目内置）
    │   └── agents_public.yaml      # 公共配置（管理员可修改）
    ├── loader.py             # 配置加载器
    └── service.py            # 配置服务
```

### 2.2 目录功能说明

| 目录 | 职责 | 关键文件 | 说明 |
|------|------|----------|----------|
| `api/` | API 层（对外接口 + 前后端连接） | tasks.py, reports.py, websocket.py, websocket_manager.py | websocket_manager.py 管理前后端连接 |
| `manager/` | 管理层（任务 + 服务 + 基础设施） | task_manager.py, report_service.py, database.py | 任务管理、业务服务、基础设施 |
| `workflow/` | 工作流 + 四阶段智能体（整体） | scheduler.py, state.py, phase1-4/ | 调度文件直接调度各阶段智能体 |
| `tools/` | 工具模块（MCP 连接） | mcp/connector.py, mcp/tool_filter.py | 只有 phase1 可调用，phase4 未来可扩展 |
| `models/` | 数据模型层 | state.py | 工作流状态模型 |
| `config/` | 配置管理层 | agents/*.yaml, loader.py, service.py | 智能体配置管理 |

## 六、数据流向图

### 6.1 前端数据流

```
用户操作
    ↓
组件 (Components)
    ↓
Composables (逻辑处理)
    ↓
API (api/*.ts)
    ↓
Stores (状态更新)
    ↓
组件重新渲染
```

### 6.2 后端数据流（v2.0 精简版）

```
API 请求
    ↓
api/*.py (路由层)
    ↓
manager/*.py (管理层：任务 + 服务 + 基础设施)
    ↓
phases/workflow_scheduler.py (调度层)
    ↓
phases/phase1-4/runner.py (执行层)
    ↓
phases/websocket_manager.py (推送层)
    ↓
WebSocket 实时推送
```

**数据流优化说明**：

| 层级 | 原目录 | 新目录 | 优化效果 |
|------|--------|--------|----------|
| 路由层 | api/ | api/ | 合并 websocket.py |
| 管理层 | manager/ + services/ + infra/ | manager/ | 减少跨层调用 |
| 调度层 | scheduler/ | phases/ | 调度器内聚到阶段层 |
| 执行层 | phases/ | phases/ | 保持不变 |
| 推送层 | pusher/ | phases/ | 推送逻辑内聚到阶段层 |

## 七、优化对比总结

### 7.1 目录数量对比

| 版本 | 目录数量 | 主要目录 |
|------|----------|----------|
| v1.0 | 10+ | api, services, manager, scheduler, phases, pusher, websocket, tools, infra, models, config |
| v2.0 | 6 | api, manager, phases, tools, models, config |

### 7.2 层级深度对比

| 版本 | 最大层级深度 | 调用链路复杂度 |
|------|--------------|----------------|
| v1.0 | 7 层 | api → services → manager → scheduler → phases → pusher |
| v2.0 | 4 层 | api → manager → phases |

### 7.3 优化收益

| 维度 | 优化前 | 优化后 | 收益 |
|------|--------|--------|------|
| 目录数量 | 10+ | 6 | 减少 40% |
| 层级深度 | 7 层 | 4 层 | 减少 43% |
| 单文件目录 | 3 个 | 0 个 | 消除单文件目录 |
| 跨层调用 | 频繁 | 最小化 | 降低复杂度 |
