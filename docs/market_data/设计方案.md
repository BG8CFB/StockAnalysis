## 一、模块定位与目标

### 1.1 模块定位

股票数据源模块（`market_data`）是一个**全局数据服务模块**，负责：

- 管理员统一配置 A股/美股/港股 数据源
- 系统定时拉取数据并存储到数据库
- 为所有业务模块提供本地数据查询服务
- 数据过期时自动触发接口更新
- 支持多数据源降级（主数据源失败自动切换）

### 1.2 核心目标

| 目标 | 描述 |
|------|------|
| **统一管理** | 管理员配置数据源，所有用户共享使用 |
| **本地化存储** | 数据存储在本地数据库，用户请求不从外部接口获取 |
| **自动更新** | 定时任务自动拉取最新数据，过期数据自动触发更新 |
| **多数据源降级** | 主数据源失败时自动切换备用数据源 |
| **高可用性** | 分布式锁、限流、缓存等多重保障机制 |

### 1.3 数据流程

```
┌─────────────────────────────────────────────────────────┐
│ 数据流程图                                               │
└─────────────────────────────────────────────────────────┘

管理员配置数据源（一次性）
    ↓
系统定时拉取数据（定时任务）
    ├─ A股：每天收盘后拉取日K线
    ├─ 美股：每天收盘后拉取日K线
    └─ 港股：每天收盘后拉取日K线
    ↓
数据存入数据库（全局共享）
    ├─ stock_quotes（行情数据）
    ├─ stock_financials（财务数据）
    └─ stock_companies（公司信息）
    ↓
用户请求数据（从本地数据库读取）
    ├─ 数据存在且未过期 → 直接返回
    └─ 数据不存在或已过期 → 触发接口更新后返回
```

### 1.4 适用场景

- **TradingAgents 智能体分析**：提供股票行情、财报、公司信息等数据
- **智能选股模块**：批量获取多只股票数据
- **AI 问股模块**：查询股票信息
- **仪表板模块**：展示市场概况和个股数据

> - [美股数据源](./美股数据源/) - Yahoo Finance、AkShare、Alpha Vantage 接口文档
> - [港股数据源](./港股数据源/) - Yahoo Finance、AkShare 接口文档
> - [数据统一存储方案](./A股数据源/A股数据统一存储方案.md) - 跨数据源字段映射与标准化

**数据源概览**：

|------|---------|-----------|-----------|
| A股 | TuShare Pro | AkShare | - |
| 美股 | Yahoo Finance | AkShare | Alpha Vantage |
| 港股 | Yahoo Finance | AkShare | iTick（暂不实现） |

### 2.2 数据获取策略

#### 2.2.1 定时拉取（主动）

**策略**：系统定时任务自动拉取数据

```
A股数据更新时机：
├─ 日K线：每天 15:30（收盘后）
├─ 实时行情：交易时间内每5分钟
└─ 财务报表：每天 16:00 检查更新

美股数据更新时机：
├─ 日K线：每天 05:00（收盘后，北京时间）
├─ 实时行情：交易时间内每1分钟
└─ 财务报表：每天 06:00 检查更新

港股数据更新时机：
├─ 日K线：每天 16:30（收盘后）
├─ 实时行情：交易时间内每5分钟
└─ 财务报表：每天 17:00 检查更新
```

#### 2.2.2 按需更新（被动）

**触发条件**：
- 用户请求的数据不存在
- 数据已过期（超过TTL时间）

**流程**：
```
用户请求数据
    ↓
检查数据库
    ├─ 数据存在且未过期 → 直接返回
    └─ 数据不存在或已过期 → 触发接口更新
        ├─ 调用数据源接口
        ├─ 数据验证
        ├─ 存入数据库
        └─ 返回给用户
```

---

### 2.3 数据过期策略（TTL）

根据**天级预测**场景设计，不需要秒级/分钟级实时性。

| 数据类型 | TTL | 存储范围 | 更新策略 | 存储表 |
|---------|-----|---------|---------|--------|
| **当日行情** | **1周** | **仅自选股** | 定时任务每天清理过期数据 | stock_quotes |
| **历史日K线** | **1周** | **仅自选股** | 写入前清理1周前的数据 | stock_klines |
| **分钟行情** | **1周** | **仅自选股** | 定时任务每天清理过期数据 | stock_minutes |
| 财务报表 | 90天 | 全市场 | 季度检查更新 | stock_financials |
| 公司信息 | 30天 | 全市场 | 每月检查更新 | stock_companies |
| 历史新闻 | 1年 | 全市场 | 定期清理 | stock_news |
| 技术指标 | - | - | 工具实时计算，不存储 | - |

**重要说明**：
- **日线/分钟行情数据**：公共数据库仅存储用户自选股的数据，保留1周
- **实时日K数据需求**：用户需从自己配置的付费接口（如Tushare Pro）实时获取，系统不同步全市场日K数据
- **设计原因**：日线数据主要用于实时短线分析，非自选股无需长期存储
- **技术指标**：由工具模块在需要时实时计算，不由数据源模块预存储

**TTL判断逻辑**：
```
数据是否过期：

场景1：历史K线、历史新闻
└─ 永不过期，一旦入库就不再更新

场景2：当日行情
└─ 如果（当前日期 != 数据日期）→ 过期

场景3：财务报表、公司信息
└─ 如果（当前时间 - fetched_at）> TTL → 过期

过期数据处理：
├─ 过期但不删除，保留历史记录
└─ 触发接口更新，获取最新数据
```

## 三、双通道数据流设计

### 3.1 设计背景

为了满足不同用户对数据实时性和成本的需求，系统设计了**双通道数据流**：

- **通道1（直接返回）**：数据直接返回给业务模块，不存储，不限流
- **通道2（存入数据库）**：数据存入系统数据库，按股票+数据类型+1分钟限流

### 3.2 双通道工作原理

#### 3.2.1 场景1：用户使用免费数据源（项目信息源）

```
┌─────────────────────────────────────────────┐
│ 用户选择"项目信息源"                         │
└─────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────┐
│ 从系统数据库读取数据                         │
│ - Tushare（免费额度）                        │
│ - AkShare（完全免费）                        │
│ - 数据可能略有延迟（通常<1天）               │
└─────────────────────────────────────────────┘
                    ↓
            返回给业务模块使用
```

#### 3.2.2 场景2：用户使用个人付费数据源（实时信息源）

```
┌─────────────────────────────────────────────┐
│ 用户选择"实时信息源"                         │
│ （用户自己配置了付费API Key）                │
└─────────────────────────────────────────────┘
                    ↓
        调用用户配置的付费接口
                    ↓
        ┌───────────┴───────────┐
        ↓                       ↓
┌───────────────┐       ┌───────────────┐
│  通道1        │       │  通道2        │
│  直接返回     │       │  存入数据库   │
│  不限流       │       │  限流控制     │
└───────────────┘       └───────────────┘
        ↓                       ↓
   直接返回给工具模块     写入系统数据库
   用于AI分析             供其他用户使用
```

### 3.3 通道2的限流策略

#### 3.3.1 限流粒度

按 **股票代码 + 数据类型** 进行限流：

| 限流键 | 说明 | 示例 |
|--------|------|------|
| `symbol + data_type` | 同一股票的同一数据类型 | `000001.SZ + quote`（行情数据）<br>`000001.SZ + financial`（财务数据）<br>`000001.SZ + company`（公司信息） |

#### 3.3.2 时间窗口

**1分钟时间窗口**：
- 如果某只股票的某种数据类型在1分钟内已经写过
- 后续的写入请求会被拒绝
- 1分钟后允许再次写入

#### 3.3.3 并发控制场景

**场景：A用户和B用户同时分析000001.SZ**

```
时间线：
├─ T0秒：A用户和B用户同时发起分析（都使用实时信息源）
├─ T0.1秒：两人都调用用户配置的API接口（通道1，不限流，都调用成功）
├─ T0.5秒：两人都获取到数据
│
├─ 通道1（直接返回，不限流）：
│   ├─ A用户的数据 → 工具模块 → AI分析
│   └─ B用户的数据 → 工具模块 → AI分析
│   （并行执行，互不影响）
│
└─ 通道2（存入数据库，限流）：
    ├─ A用户尝试写入000001.SZ的quote数据
    │   ├─ 检查限流记录：不存在
    │   ├─ 允许写入 → 写入成功
    │   └─ 记录限流：000001.SZ + quote（过期时间1分钟）
    │
    └─ B用户尝试写入000001.SZ的quote数据
        ├─ 检查限流记录：存在且未过期
        ├─ 拒绝写入 → 数据丢弃
        └─ B用户的通道2数据不写入数据库
```

**时间差场景**：
```
时间线：
├─ T0秒：A用户分析000001.SZ → 写入成功 → 记录限流（过期时间T0+60秒）
├─ T20秒：B用户分析000001.SZ → 检查限流 → 在1分钟内 → 拒绝写入
├─ T70秒：C用户分析000001.SZ → 检查限流 → 已超过1分钟 → 允许写入
```

### 3.4 数据一致性说明

#### 3.4.1 盘中数据 vs 收盘数据

**盘中数据**：
- 交易时间内获取的数据（如14:30）
- 数据字段 `is_complete: false`（未收盘）
- 可能是未完成的当日行情

**收盘数据**：
- 收盘后获取的完整数据（15:30后）
- 数据字段 `is_complete: true`（已收盘）
- 完整的日K线数据

**写入数据库时**：
- 用户在14:30调用接口获取盘中数据 → `is_complete: false`
- 定时任务在15:30获取收盘数据 → `is_complete: true`
- 用户使用"项目信息源"时优先返回 `is_complete: true` 的数据

#### 3.4.2 数据允许差异

**原则**：实时信息源和项目信息源的数据可能不同，这是允许的。

- 用户选择"实时信息源"：获取的是接口实时返回的数据（可能是盘中数据）
- 用户选择"项目信息源"：获取的是数据库中的数据（可能是昨天的收盘数据）
- 单次任务只能选择一个数据源

### 3.5 降级逻辑

**用户使用"实时信息源"时的降级流程**：

```
用户发起分析任务（选择实时信息源）
    ↓
检查用户是否配置了该市场的实时数据源
    ↓
├─ 未配置 → 使用"项目信息源"（从数据库获取）
│
└─ 已配置 → 调用用户配置的接口
    ├─ 调用成功
    │   ├─ 通道1：直接返回（不限流）
    │   └─ 通道2：尝试写入数据库（限流）
    │
    └─ 调用失败 → 降级到"项目信息源"（从数据库获取）
```

## 四、用户级数据源配置

### 4.1 用户级数据源定位

**配置者**：每个用户自己
**可见性**：仅用户自己可见
**使用场景**：TradingAgents选择"实时信息源"时
**数据存储**：直接返回，不存储（通道1）；存入数据库，限流（通道2）

### 4.2 配置规则

#### 4.2.1 允许配置的付费数据源

**原则**：只展示需要付费的接口让用户配置，免费数据源不允许用户添加。

| 市场 | 数据源 | 是否需要Key | 说明 |
|------|--------|------------|------|
| A股 | Tushare Pro | ✅ 需要积分 | 用户自己的Token |
| 美股 | Alpha Vantage | ✅ 需要Key | 免费版限制较多 |
| 港股 | iTick | ✅ 需要Key | 付费数据源 ⚠️ **暂不实现** |

**不允许用户配置的免费数据源**：
- ❌ AkShare（完全免费）
- ❌ Yahoo Finance（免费API）

**原因**：免费数据源使用系统公共配置即可，避免重复配置。

#### 4.2.2 配置粒度

按市场分别配置（A股、美股、港股）：

**配置结构**：
```yaml
user_data_sources:
  A_STOCK:
    - source_id: "tushare_pro"
      api_key: "用户自己的Tushare Token"
      enabled: true
      priority: 1

  US_STOCK:
    - source_id: "alpha_vantage"
      api_key: "用户自己的Alpha Vantage Key"
      enabled: true
      priority: 1

  HK_STOCK:
    - source_id: "itick"
      api_key: "用户自己的iTick Key"
      enabled: true
      priority: 1
```

### 4.3 与系统级数据源的区别

| 对比项 | 系统级数据源（公共） | 用户级数据源（个人） |
|--------|-------------------|-------------------|
| **配置者** | 超级管理员 | 每个用户自己 |
| **可见性** | 所有用户共享 | 仅用户自己可见 |
| **API Key** | 公共Key | 用户自己的Key |
| **数据源类型** | 免费为主（Tushare免费额度、AkShare） | 仅付费接口 |
| **数据存储** | 存入数据库，所有用户共享 | 通道1：直接返回<br>通道2：存入数据库（限流） |
| **使用场景** | 所有模块（AI问答、选股、TradingAgents项目信息源） | TradingAgents（实时信息源） |
| **成本** | 系统承担 | 用户自己承担 |

### 4.4 用户级数据源的价值

**为用户提供的价值**：
- 获取最新实时数据（可能是盘中数据）
- 不受系统公共数据源的限流限制
- 数据存入数据库后，其他使用"项目信息源"的用户也能获取到相对实时的数据

**为系统提供的价值**：
- 减轻系统公共数据源的API调用压力
- 用户付费数据源的数据进入数据库，丰富数据维度
- 用户配置了实时数据源后，分析任务不消耗系统配额

### 5.2 数据源适配器说明

#### 5.2.1 适配器设计原则

**为什么按市场分目录**：
- 不同市场的数据特点完全不同（A股/美股/港股）
- 每个市场可能有3-4个数据源，每个适配器文件负责一个数据源
- 便于管理和维护，新增数据源时只需添加对应市场的适配器文件

**目录组织逻辑**：
```
sources/
├─ a_stock/          # A股数据源目录
│  ├─ tushare_adapter.py    # Tushare 适配器（500+ 行）
│  └─ akshare_adapter.py    # AkShare 适配器（400+ 行）
├─ us_stock/         # 美股数据源目录
│  ├─ yahoo_adapter.py       # Yahoo Finance 适配器
│  ├─ akshare_adapter.py     # AkShare（美股）适配器
│  └─ alphavantage_adapter.py # Alpha Vantage 适配器
└─ hk_stock/         # 港股数据源目录
   ├─ yahoo_adapter.py       # Yahoo Finance 适配器
   ├─ akshare_adapter.py     # AkShare（港股）适配器
   └─ itick_adapter.py       # iTick 适配器 ⚠️ 暂不实现
```

**每个适配器的职责**：
1. 封装对应数据源的API调用
2. 实现字段映射（原始字段 → 标准字段）
3. 实现数据类型转换
4. 实现单位标准化（手→股，千元→元）
5. 实现股票代码标准化
6. 实现数据验证

#### 5.2.2 适配器接口规范

**所有适配器必须实现的方法**：

```
DataSourceAdapter（抽象基类）
├─ get_daily_quote(symbol, date)        # 获取单日行情
├─ get_klines(symbol, start, end)       # 获取历史K线
├─ get_financials(symbol, period)       # 获取财务报表
├─ get_company_info(symbol)             # 获取公司信息
├─ get_news(symbol, start, end)         # 获取新闻
└─ normalize_data(raw_data, data_type)  # 数据标准化
```

**标准化流程**：
```
原始数据 → 字段映射 → 类型转换 → 单位标准化 → 代码标准化 → 标准数据
```

## 六、统一数据模型与标准化

### 4.1 设计原则

**核心问题**：不同数据源返回的数据格式、字段名、数据类型各不相同，需要统一为标准格式后才能存入数据库。

**设计原则**：
1. **以 Tushare 为主要参考标准**：Tushare 数据最全面，字段定义最规范
2. **统一字段命名**：采用小写下划线命名法（snake_case）
3. **统一数据类型**：价格用 float，成交量用 int，日期用 YYYY-MM-DD 字符串
4. **统一股票代码格式**：`{代码}.{市场后缀}`，如 `600000.SH`、`AAPL.US`、`0700.HK`

---

### 4.2 股票代码标准化规范

#### 4.2.1 标准格式定义

```
标准格式：{股票代码}.{市场后缀}

市场后缀规范：
├─ A股（上海）：.SH
├─ A股（深圳）：.SZ
├─ A股（北交所）：.BJ
├─ 美股：.US
├─ 港股：.HK
```

#### 4.2.2 各数据源代码格式转换

| 数据源 | 原始格式 | 标准格式 | 转换规则 |
|--------|---------|---------|---------|
| **Tushare** | `600000.SH` | `600000.SH` | 无需转换（已是标准格式） |
| **AkShare** | `sh600000` | `600000.SH` | 去掉前缀 sh/sz，添加后缀 .SH/.SZ |
| **Yahoo Finance** | `600000.SS` | `600000.SH` | `.SS` → `.SH`，`.SZ` 保持 |
| **Yahoo Finance（美股）** | `AAPL` | `AAPL.US` | 添加 `.US` 后缀 |
| **Yahoo Finance（港股）** | `0700.HK` | `0700.HK` | 无需转换 |

## 七、ETL 数据处理流程

### 5.1 ETL 概述

ETL（Extract-Transform-Load）是将多数据源数据标准化并存入数据库的核心流程。

```
ETL 流程概览：

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Extract   │ ──→ │  Transform  │ ──→ │    Load     │
│   (提取)     │     │   (转换)    │     │   (加载)    │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      ↓                    ↓                    ↓
从数据源获取         字段映射、清洗          存入数据库
   原始数据           格式标准化              更新索引
```

---

### 5.2 Extract（提取）阶段

```
提取流程：

1. 根据市场类型获取数据源优先级列表
   ├─ A股：[Tushare, AkShare]
   ├─ 美股：[Yahoo Finance, AkShare, Alpha Vantage]
   └─ 港股：[Yahoo Finance, AkShare, iTick]

2. 按优先级依次尝试获取数据
   ├─ 优先级1成功 → 返回原始数据
   └─ 优先级1失败 → 尝试优先级2
       ├─ 优先级2成功 → 返回原始数据
       └─ 优先级2失败 → 尝试优先级3

3. 记录数据来源
   └─ 返回：{ raw_data, source_id, fetch_time }
```

---

### 5.3 Transform（转换）阶段

#### 5.3.1 转换流程

```
转换流程：

原始数据
    ↓
步骤1：字段映射
├─ 根据数据源选择对应的映射规则
├─ 将原始字段名转换为标准字段名
└─ 处理字段缺失（使用 null 或默认值）
    ↓
步骤2：数据类型转换
├─ 价格字段 → float（保留2位小数）
├─ 成交量 → int
├─ 日期 → YYYY-MM-DD 字符串
└─ 百分比 → float（如 5.23 表示 5.23%）
    ↓
步骤3：单位标准化
├─ Tushare 成交量（手）→ 股：× 100
├─ Tushare 成交额（千元）→ 元：× 1000
└─ 其他数据源检查并统一
    ↓
步骤4：股票代码标准化
├─ 识别原始代码格式
├─ 转换为标准格式：{代码}.{后缀}
└─ 验证代码有效性
    ↓
步骤5：数据清洗
├─ 去除空白字符
├─ 处理异常值（价格为0、负数等）
└─ 处理缺失值
    ↓
步骤6：数据验证
├─ 完整性检查（必填字段）
├─ 合理性检查（价格>0，成交量>=0）
└─ 一致性检查（high>=low）
    ↓
标准化数据
```

#### 5.3.2 异常数据处理规则

| 异常类型 | 处理方式 |
|---------|---------|
| 价格为 0 | 使用前一日收盘价填充，或标记为无效 |
| 价格为负数 | 拒绝该条数据 |
| 成交量为负数 | 拒绝该条数据 |
| high < low | 拒绝该条数据 |
| 日期格式错误 | 尝试解析，失败则拒绝 |
| 必填字段缺失 | 尝试从其他数据源补充，否则拒绝 |

---

### 5.4 Load（加载）阶段

```
加载流程：

标准化数据
    ↓
步骤1：去重检查
├─ 查询数据库是否已存在相同记录
├─ 唯一键：symbol + trade_date（行情数据）
└─ 存在 → 更新；不存在 → 插入
    ↓
步骤2：写入数据库
├─ 添加元数据字段
│   ├─ data_source: 数据来源
│   ├─ fetched_at: 获取时间
│   ├─ data_quality: 数据质量评分
│   └─ expires_at: 过期时间
└─ 执行插入/更新操作
    ↓
步骤3：更新索引（自动）
    ↓
步骤4：记录操作日志
├─ 成功数量
├─ 失败数量
└─ 失败原因
```

### 6.1 同步类型概览

| 同步类型 | 触发方式 | 同步内容 | 数据范围 |
|---------|---------|---------|---------|
| **定时同步** | 定时任务 | 轻量数据 | 所有股票 |

### 6.2 定时同步策略

**同步目标**：
- **A股**：全市场覆盖（基础行情+财务）
- **美股/港股**：仅覆盖**自选股**、**持仓股**及**核心指数**

**同步内容**：
- ✅ 当日行情数据（开高低收量）
- ✅ 公司基本信息（变化时更新）
- ✅ 最新财务指标
- ❌ 历史日K线（仅自选股同步）
- ❌ 历史新闻（仅自选股同步）

```
定时同步时间表：

A股市场（全量）：
├─ 15:30 - 同步全市场当日行情
├─ 16:00 - 检查并更新公司信息（每周一次）
└─ 16:30 - 检查并更新财务指标（每日）

美股/港股市场（自选股）：
├─ 收盘后 - 同步自选股当日行情
├─ 收盘后 - 检查自选股财务更新
└─ 实时 - 用户查看非自选股时触发实时获取
```

---

### 6.3 自选股同步策略

**触发时机**：用户添加股票到自选列表

**同步内容**：
- ✅ 公司基本信息
- ✅ 历史日K线（时间范围由用户配置，默认1年）
- ✅ 历史财务报表（近4期）
- ✅ 历史新闻（时间范围由用户配置，默认1年）
- ✅ 技术指标计算

**用户可配置项**：

| 配置项 | 默认值 | 可选范围 | 说明 |
|--------|-------|---------|------|
| 日K线历史范围 | 1年 | 3个月/6个月/1年/2年/3年 | 同步多长时间的历史K线 |
| 历史新闻范围 | 1年 | 3个月/6个月/1年/2年 | 同步多长时间的历史新闻 |
| 财务报表期数 | 4期 | 4期/8期/12期 | 同步多少期财务报表 |

---

### 6.4 按需同步策略

**触发时机**：
1. 用户请求的数据在数据库中不存在
2. 用户请求的数据已过期（超过 TTL）
3. AI 工具调用需要数据但数据缺失

**这就是「缓存命中」概念**：数据库作为数据中转站，首次请求触发同步，后续请求直接复用。

```
按需同步流程：

用户/工具请求数据
    ↓
检查数据库
├─ 数据存在且未过期 → 直接返回（缓存命中）
└─ 数据不存在或已过期 → 触发同步
    ↓
获取分布式锁
├─ 获取成功 → 执行同步
│   ├─ 调用数据源API
│   ├─ ETL处理
│   ├─ 存入数据库
│   └─ 释放锁
└─ 获取失败 → 等待2秒后重新检查数据库
    └─ 其他请求已完成同步，直接返回
    ↓
返回数据给用户/工具
```

## 十、核心业务逻辑设计

### 8.1 数据源配置模型

#### 8.1.1 配置结构

```yaml
数据源配置：
  source_id: "tushare"
  market: "A_STOCK"
  enabled: true
  priority: 1  # 优先级（数字越小优先级越高）

  # 数据源配置
  config:
    api_key: "encrypted_api_key"  # 加密存储
    base_url: "https://api.tushare.pro"
    timeout: 30  # 超时时间（秒）

  # 限流配置
  rate_limit:
    max_requests_per_minute: 1000
    max_requests_per_day: 10000

  # 数据类型支持
  supported_data_types:
    - "quote"      # 行情
    - "history"     # 历史K线
    - "financials"  # 财务报表
    - "company"     # 公司信息

  # 元数据
  metadata:
    created_at: datetime
    updated_at: datetime
    last_test_status: "success"  # success | failed
    last_tested_at: datetime
```

#### 8.1.2 多数据源优先级

```
每个市场可配置多个数据源，按优先级排序：

A股数据源示例：
├─ 1. Tushare（优先级1，主数据源）
└─ 2. AkShare（优先级2，备用数据源）

数据获取流程：
├─ 尝试优先级1：Tushare
│   ├─ 成功 → 返回数据
│   └─ 失败 → 降级到优先级2
└─ 尝试优先级2：AkShare
    ├─ 成功 → 返回数据
    └─ 失败 → 返回错误
```

---

### 8.2 数据获取流程

#### 8.2.1 按需更新流程

```
用户请求股票数据
    ↓
步骤1：检查数据库
├─ 查询条件：股票代码、市场、日期
├─ 判断数据是否存在
└─ 判断数据是否过期
    ↓
步骤2：数据存在且未过期
└─ 直接返回（最快）
    ↓
步骤3：数据不存在或已过期
├─ 尝试获取分布式锁（防止并发重复获取）
├─ 获取到锁 → 调用数据源接口
│   ├─ 按优先级尝试数据源
│   ├─ 数据验证
│   ├─ 标准化字段
│   ├─ 存入数据库
│   └─ 释放锁
└─ 未获取到锁 → 等待2秒，再次检查数据库
    └─ 其他用户已更新，直接返回
```

#### 8.2.2 分布式锁工作原理

**防止并发重复获取**：

```
场景：100个用户同时请求 AAPL 数据（缓存已过期）

没有分布式锁：
└─ 100个用户都调用API获取数据
    └─ 浪费99次API调用

有分布式锁：
├─ 用户1获取到锁 → 调用API获取数据
├─ 用户2-100未获取到锁 → 等待2秒
├─ 用户1获取完数据，释放锁
└─ 用户2-100等待结束，从数据库获取数据
    └─ 只调用1次API，节省99次
```

**锁的配置**：
| 参数 | 值 | 说明 |
|------|-----|------|
| 键格式 | `lock:data_fetch:{market}:{symbol}` | 例如：lock:data_fetch:A_STOCK:AAPL |
| 超时时间 | 30秒 | 防止死锁 |
| 等待时间 | 2秒 | 未获取锁的用户等待时间 |

### 8.3 数据验证机制

#### 8.3.1 验证流程

```
从API获取的原始数据
    ↓
步骤1：字段标准化
├─ 不同数据源字段名不同
├─ 统一为标准字段名
└─ 示例：Open/OPEN → open
    ↓
步骤2：完整性检查
├─ 检查必填字段
├─ 必填字段：open, high, low, close, volume
└─ 缺少 → 拒绝数据，尝试备用数据源
    ↓
步骤3：合理性检查
├─ 价格必须 > 0
├─ 成交量必须 >= 0
└─ 不合理 → 拒绝数据
    ↓
步骤4：一致性检查
├─ high >= low
├─ high >= open, high >= close
├─ low <= open, low <= close
└─ 不一致 → 拒绝数据
    ↓
步骤5：数据质量评分
├─ 完整性：40分
├─ 合理性：30分
└─ 一致性：30分
    ↓
验证通过的数据
├─ 质量评分 > 70分 → 存入数据库
└─ 质量评分 <= 70分 → 记录警告，但仍然存入
```

#### 8.3.2 验证规则详解

**完整性检查**：
```
必填字段：
├─ open（开盘价）
├─ high（最高价）
├─ low（最低价）
├─ close（收盘价）
└─ volume（成交量）

可选字段：
├─ amount（成交额）
├─ pe_ratio（市盈率）
└─ turnover_rate（换手率）
```

**一致性检查**：
```
规则1：high >= low
└─ 最高价必须 >= 最低价

规则2：high >= open 且 high >= close
└─ 最高价必须 >= 开盘价和收盘价

规则3：low <= open 且 low <= close
└─ 最低价必须 <= 开盘价和收盘价
```

**数据质量评分**：
```
总分：100分

维度1：完整性（40分）
└─ 必填字段完整率 × 40

维度2：合理性（30分）
└─ 价格、成交量合理性

维度3：一致性（30分）
└─ 价格关系一致性

评分等级：
├─ 90-100分：高质量
├─ 70-89分：中等质量
└─ <70分：低质量（记录警告）
```

### 8.4 限流策略

#### 8.4.1 为什么需要限流？

**场景**：数据源有调用频率限制

```
Yahoo Finance 免费API：
└─ 限制：2000次/小时
    └─ 超过 → IP被封锁，1小时无法使用

Tushare 免费API：
└─ 限制：1000次/分钟，10000次/天
    └─ 超过 → 返回错误，无法获取数据
```

#### 8.4.2 限流算法：令牌桶

**原理**：
```
令牌桶配置：
├─ 桶容量：100个令牌
├─ 补充速率：1个/秒
└─ 消耗规则：1次请求消耗1个令牌

工作流程：
├─ 正常情况：每秒1个请求（消耗1个，补充1个，保持平衡）
├─ 突发情况：可以瞬间发送100个请求（桶满了）
└─ 突发后：要等100秒才能恢复到100个令牌
```

**优点**：
- ✅ 允许突发流量（批量任务）
- ✅ 平滑限流（不会在时间边界突然重置）
- ✅ 适合定时任务场景

## 十二、技术指标计算

### 12.1 计算策略

**决策**：技术指标由**工具模块实时计算**，数据源模块不预存储。

**原因**：
1. **灵活性**：工具可以根据需求计算任意参数的指标
2. **数据新鲜度**：实时计算保证指标基于最新数据
3. **存储效率**：避免存储大量可计算的指标数据
4. **解耦设计**：数据源模块专注数据获取，指标计算由工具模块负责

**计算时机**：
- TradingAgents 工具调用时实时计算
- 基于查询到的K线数据动态生成

---

### 10.2 支持的技术指标

#### 10.2.1 移动平均线（MA）

| 指标 | 计算周期 | 说明 |
|------|---------|------|
| MA5 | 5日 | 短期均线 |
| MA10 | 10日 | 短期均线 |
| MA20 | 20日 | 中期均线 |
| MA60 | 60日 | 长期均线 |
| MA120 | 120日 | 半年线 |
| MA250 | 250日 | 年线 |

**计算公式**：
```
MA(N) = (C1 + C2 + ... + CN) / N

其中：
- C1, C2, ..., CN 为最近 N 日收盘价
- N 为计算周期
```

---

#### 10.2.2 MACD 指标

| 指标 | 参数 | 说明 |
|------|------|------|
| MACD | (12, 26, 9) | 默认参数 |
| DIF | 12日EMA - 26日EMA | 快线 |
| DEA | DIF的9日EMA | 慢线 |
| MACD柱 | (DIF - DEA) × 2 | 柱状图 |

**计算公式**：
```
EMA(N) = 前一日EMA × (N-1)/(N+1) + 今日收盘价 × 2/(N+1)

DIF = EMA(12) - EMA(26)
DEA = DIF 的 9日EMA
MACD柱 = (DIF - DEA) × 2
```

---

#### 10.2.3 RSI 指标

| 指标 | 参数 | 说明 |
|------|------|------|
| RSI6 | 6日 | 短期超买超卖 |
| RSI12 | 12日 | 中期超买超卖 |
| RSI24 | 24日 | 长期超买超卖 |

**计算公式**：
```
RS = N日内收盘涨幅平均值 / N日内收盘跌幅平均值
RSI = 100 - 100 / (1 + RS)

取值范围：0 ~ 100
- RSI > 70：超买区域
- RSI < 30：超卖区域
```

---

#### 10.2.4 KDJ 指标

| 指标 | 参数 | 说明 |
|------|------|------|
| K值 | 9日RSV的3日平滑 | 快速指标 |
| D值 | K值的3日平滑 | 慢速指标 |
| J值 | 3K - 2D | 超买超卖 |

**计算公式**：
```
RSV = (今日收盘价 - 9日最低价) / (9日最高价 - 9日最低价) × 100

K = 2/3 × 前一日K + 1/3 × 今日RSV
D = 2/3 × 前一日D + 1/3 × 今日K
J = 3K - 2D

取值范围：
- K, D：0 ~ 100
- J：可能超出 0 ~ 100
```

---

#### 10.2.5 布林带（BOLL）

| 指标 | 参数 | 说明 |
|------|------|------|
| 中轨 | 20日MA | 移动平均线 |
| 上轨 | 中轨 + 2倍标准差 | 压力线 |
| 下轨 | 中轨 - 2倍标准差 | 支撑线 |

**计算公式**：
```
中轨 = MA(20)
标准差 = SQRT(Σ(Ci - MA)² / 20)
上轨 = 中轨 + 2 × 标准差
下轨 = 中轨 - 2 × 标准差
```

---

### 10.3 计算流程

```
工具模块请求技术指标
    ↓
从数据库/API获取K线数据
    ↓
检查历史数据是否充足
├─ MA250 需要至少 250 条历史数据
├─ 数据不足 → 计算可用的指标，其他标记为 null
└─ 数据充足 → 计算所有指标
    ↓
实时计算各项指标
├─ MA系列（需要的历史数据最少）
├─ MACD
├─ RSI
├─ KDJ
└─ BOLL
    ↓
直接返回给工具模块使用
    ↓
计算完成（不存储到数据库）
```

**说明**：技术指标不存储到数据库，每次请求时根据K线数据实时计算。

## 十五、并发与限流策略

### 15.1 分布式锁

**目的**：防止并发请求重复获取数据

**工作流程**：
```
并发场景：100个用户同时请求 AAPL 数据

时间线：
├─ T+0秒：用户1-100同时请求数据
├─ T+0.001秒：用户1获取到分布式锁
├─ T+0.002秒：用户2-100未获取到锁，进入等待
├─ T+0.100秒：用户1调用API获取数据
├─ T+0.500秒：用户1存入数据库，释放锁
├─ T+2.002秒：用户2-100等待结束，从数据库获取数据
└─ 结果：只调用1次API，节省99次
```

**锁配置**：
| 参数 | 值 |
|------|-----|
| 键格式 | `lock:data_fetch:{market}:{symbol}` |
| 超时时间 | 30秒 |
| 等待时间 | 2秒 |

**配置**：

## 十六、错误处理与降级

### 16.1 多数据源降级策略

**降级流程**：
```
主数据源失败
    ↓
尝试备用数据源1
    ├─ 成功 → 返回数据
    └─ 失败 → 进入下一步
        ↓
尝试备用数据源2
    ├─ 成功 → 返回数据
    └─ 失败 → 返回错误
```

**降级触发条件**：
- ✅ 网络超时（>30秒）
- ✅ 接口返回错误（HTTP 5xx）
- ✅ 数据解析失败
- ✅ 数据验证失败

**降级日志记录**：
```
日志内容：
├─ 尝试的数据源
├─ 失败原因
├─ 降级到哪个数据源
└─ 最终结果
```

---

### 16.2 错误分类与处理

| 错误类型 | 处理方式 | 是否重试 |
|---------|---------|---------|
| 网络超时 | 降级到备用数据源 | ✅ 是 |
| API限流（429） | 等待后重试 | ✅ 是 |
| 数据验证失败 | 降级到备用数据源 | ❌ 否 |
| 所有数据源失败 | 返回明确错误 | ❌ 否 |

---

### 16.3 场景化降级策略

#### 16.3.1 定时任务场景（数据同步）

**特点**：后台自动执行，用户不直接等待结果

```
定时任务调用接口获取数据
    ↓
调用主数据源接口
    ↓
├─ 成功
│   ├─ 更新状态: healthy
│   ├─ 清除失败计数
│   ├─ 记录性能指标
│   └─ 如果之前是降级状态，记录恢复事件
│
└─ 失败
    ├─ 失败计数 +1
    ├─ 记录错误信息
    │
    ├─ 失败次数 < 3
    │   ├─ 立即重试（最多3次）
    │   └─ 更新状态: degraded（临时降级）
    │
    └─ 失败次数 >= 3
        ├─ 标记状态: unavailable
        ├─ 记录事件: api_failed
        │
        ├─ 尝试降级到备用数据源
        │   ├─ 备用数据源1 成功
        │   │   ├─ 更新 fallback_info
        │   │   ├─ 记录事件: fallback
        │   │   └─ 返回数据
        │   │
        │   └─ 备用数据源1 失败 → 尝试备用数据源2
        │       └─ 依此类推...
        │
        └─ 所有备用数据源都失败
            ├─ 标记整个数据类型为 failed
            └─ 返回错误
```

#### 16.3.2 手动触发/工具调用场景

**特点**：用户主动操作或工具调用，用户在等待结果

```
用户手动触发 或 TradingAgents 工具调用
    ↓
调用主数据源接口
    ↓
├─ 成功 → 返回数据，更新状态
│
└─ 失败
    ├─ 不重试！（用户在等待结果）
    ├─ 记录错误信息
    │
    └─ 立即尝试降级到备用数据源
        ├─ 备用成功 → 返回数据，记录降级事件
        └─ 备用失败 → 返回错误，标记 failed
```

#### 16.3.3 手动重试逻辑（用户点击"重试"按钮）

```
用户点击"重试"按钮
    ↓
仅重试失败的主数据源接口
    ↓
├─ 成功
│   ├─ 清除失败计数
│   ├─ 更新状态: healthy
│   ├─ 如果当前使用备用数据源，切换回主数据源
│   └─ 记录事件: recovered
│
└─ 失败
    ├─ 失败计数 +1
    ├─ 更新错误信息
    └─ 保持 unavailable 状态
```

#### 16.3.4 多接口数据类型的降级策略

**场景示例**：日线数据需要 Tushare 的 2 个接口

| 接口状态 | 处理策略 | 最终状态 |
|---------|---------|---------|
| 全部正常 | 使用 Tushare | ✅ healthy |
| pro_bar 正常，adj_factor 失败 | 尝试使用 daily（基础日线）补偿 | ⚠️ degraded（部分数据） |
| pro_bar 失败 | 立即降级到 AkShare（有完整数据） | ⚠️ degraded（降级） |
| 全部失败且 AkShare 也失败 | 标记 unavailable | ❌ unavailable |

**补偿逻辑**：
```

1. 尝试使用 Tushare 的 daily 接口（基础日线，不复权）
   ├─ 成功 → 数据可用，但缺少复权信息，标记为"部分异常"
   └─ 失败 → 进入步骤2

2. 降级到 AkShare
   ├─ AkShare 有复权数据，数据完整性较好
   ├─ 标记状态为"已降级"
   └─ 记录降级原因

3. 如果 AkShare 也失败
   └─ 标记为不可用，显示重试按钮
```

## 十九、总结

### 19.1 核心设计原则

1. **管理员统一配置**：所有数据源由管理员配置，用户无需关心
2. **本地化存储**：数据存储在本地数据库，用户请求不直接调用外部API
3. **分层同步策略**：
   - 定时同步：轻量数据（当日行情、公司信息）
   - 自选股同步：完整数据（历史K线、新闻、财报）
   - 按需同步：数据缺失或过期时触发
4. **缓存命中机制**：数据库作为数据中转站，首次请求触发同步，后续复用
5. **多数据源降级**：主数据源失败自动切换备用数据源
6. **并发控制**：分布式锁防止重复获取
7. **数据标准化**：统一数据模型，ETL 流程处理
8. **技术指标后端计算**：保证一致性和复用性

---

### 16.2 关键技术点

| 技术点 | 说明 |
|--------|------|
| **统一数据模型** | 以 Tushare 为参考标准，定义标准字段 |
| **ETL 流程** | Extract-Transform-Load 三阶段数据处理 |
| **字段映射** | Tushare/AkShare → 标准字段的转换规则 |
| **股票代码标准化** | 统一为 {代码}.{后缀} 格式 |
| **分布式锁** | 防止并发重复获取数据 |
| **令牌桶限流** | 保护API配额 |
| **数据验证** | 完整性、合理性、一致性三层验证 |
| **技术指标计算** | MA/MACD/RSI/KDJ/BOLL 后端计算并存储 |
| **多数据源降级** | 按优先级自动切换数据源 |
