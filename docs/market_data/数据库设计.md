## 一、数据库概述

### 1.1 数据库选型

本项目使用 **MongoDB** 作为主要数据库，原因：
- 文档型数据库，适合存储股票行情、财务报表等结构化数据
- 支持水平扩展，便于应对海量历史数据
- 灵活的Schema设计，便于添加新字段
- 内置分片和副本集支持，保证高可用性

### 1.2 数据库命名规范

**数据库名**：`stock_analysis`

**集合命名规范**：
- 使用小写字母和下划线
- 采用 `模块_实体` 的命名方式
- 示例：`stock_quotes`、`user_data_sources`、`system_data_sources`

### 1.3 字段命名规范

**统一使用小写下划线命名法（snake_case）**：
- ✅ `trade_date`、`data_source`、`api_key`
- ❌ `tradeDate`、`dataSource`、`apiKey`

### 1.4 数据类型规范

| 数据类型 | MongoDB类型 | 说明 |
|---------|------------|------|
| 字符串 | String | 股票代码、日期、名称等 |
| 数值 | NumberDecimal | 价格、金额等高精度数值 |
| 整数 | Int32/Int64 | 成交量、数量等 |
| 日期 | Date | 时间戳 |
| 布尔 | Boolean | 开关状态 |
| 对象 | Object | 嵌套对象 |
| 数组 | Array | 列表数据 |
| 对象ID | ObjectId | 关联ID |

## 二、多市场存储策略设计

### 2.1 设计目标

本项目需要支持多市场（A股、美股、港股）的数据存储，核心设计目标：
- **数据量可控**：确保数据量在合理范围内，避免过度增长
- **查询性能**：单股票查询场景下，保证毫秒级响应
- **维护简单**：避免过度复杂的表结构，降低开发和运维成本
- **业务匹配**：符合单股票分析为主、多市场数据关联度低的业务特点

### 2.3 存储策略决策

#### 2.3.1 最终方案：统一表 + market字段

**核心决策**：采用**统一表设计**，所有市场数据存储在同一张表中，通过 `market` 字段区分市场。

**表结构示例**：
```javascript
// stock_quotes（统一存储三市场）
{
  _id: ObjectId("..."),
  symbol: "600000.SH",        // 股票代码
  market: "A_STOCK",          // 市场标识
  trade_date: "2026-01-05",   // 交易日期
  open: 10.50,
  close: 10.70,
  // ...
}

{
  _id: ObjectId("..."),
  symbol: "AAPL",             // 美股代码
  market: "US_STOCK",         // 市场标识
  trade_date: "2026-01-05",
  open: 180.20,
  close: 182.50,
  // ...
}
```

#### 2.3.2 选择统一表的理由

**理由1：数据量可控**
- 三市场合并后的日线行情数据量约437.5万条/年
- MongoDB单表可轻松处理千万级数据
- 在1年保留策略下，数据总量远未达到性能瓶颈

**理由2：业务逻辑统一**
- 同类型数据（如日线行情）的业务逻辑高度相似
- 统一表便于实现统一的数据清理、限流、技术指标计算逻辑
- 避免为每个市场重复实现相同功能

**理由3：查询性能优化**
- 主查询场景：单股票查询 `WHERE symbol = ? AND trade_date >= ? AND trade_date <= ?`
- 使用复合索引 `(symbol, trade_date)` 或 `(market, symbol, trade_date)`
- 单表查询性能不会因数据量增加而显著下降

**理由4：简化开发与运维**
- 代码中无需判断市场类型，统一处理
- 减少表数量，降低维护成本
- 便于未来扩展新市场（如加英股、日股等）

**理由5：跨市场查询需求**
- 虽然当前主要是单股票分析，但未来可能有跨市场对比需求
- 统一表便于实现"查询同一公司在A股和港股的表现"等场景

#### 2.3.3 按市场分离表的缺陷分析

**方案对比**（按市场分离表）：
```
stock_quotes_a_stock    // A股行情
stock_quotes_us_stock   // 美股行情
stock_quotes_hk_stock   // 港股行情
```

**主要缺陷**：
1. **代码复杂度高**：需要为每个市场单独实现数据获取、清理、限流逻辑
2. **维护成本高**：3倍的表数量，3倍的索引，3倍的清理任务
3. **跨市场查询困难**：需要UNION查询多个表，性能差
4. **过度设计**：数据量远未达到需要分表的程度

### 2.4 差异化保留策略

基于数据量特点和业务场景，采用差异化保留策略：

| 数据类型 | 存储表 | 保留时长 | 存储范围 | 策略说明 |
|---------|-------|---------|---------|---------|
| **日线行情** | `stock_quotes` | **1周** | **仅自选股** | 公共数据库只存用户自选的股票日K数据 |
| **日K线+指标** | `stock_klines` | **1周** | **仅自选股** | 用于短线分析场景，技术指标由工具实时计算 |
| **分钟行情** | `stock_minutes` | **1周** | **仅自选股** | 短期缓存，避免数据量爆炸 |
| 财务报表 | `stock_financials` | 1年（4期） | 全市场定时同步 | 仅保留最新4期 |
| 公司信息 | `stock_companies` | 1年 | 全市场定时同步 | 定期更新 |
| 新闻资讯 | `stock_news` | 1年 | 全市场定时同步 | 定期清理 |
| 宏观经济 | `macro_economic` | 永久 | 全市场 | 数据量极小 |
| 股票基础信息 | `stock_info` | 永久 | 全市场定时同步 | 股票列表基础数据 |

**核心原则**：
- **行情数据（日线/分钟）**：仅存储用户自选股，保留1周
- **实时日K数据**：用户需要实时数据时，从自己配置的付费接口（如Tushare Pro）直接获取，不存储到公共数据库
- **基础数据（财务/公司/新闻）**：全市场定时同步，长期保留
- 通过 `(market, symbol, trade_date)` 复合索引保证查询性能

**日线数据获取逻辑**：
```
用户请求日线数据
    ↓
检查用户是否配置了付费数据源
    ├─ 已配置 → 从用户自有接口实时获取，直接返回（不存储）
    └─ 未配置 → 从公共数据库查询
        ├─ 数据存在（自选股）→ 直接返回
        └─ 数据不存在 → 降级到AkShare实时获取
```

### 2.5 索引优化设计

#### 2.5.1 核心索引

```javascript
// 单股票历史查询（最常见）
db.stock_quotes.createIndex({ symbol: 1, trade_date: -1 })

// 单市场查询
db.stock_quotes.createIndex({ market: 1, trade_date: -1 })

// 单股票单市场查询
db.stock_quotes.createIndex({ market: 1, symbol: 1, trade_date: -1 })
```

#### 2.5.2 查询性能保证

**场景1：查询单股票1年K线数据**
```javascript
db.stock_klines.find({
  symbol: "600000.SH",
  trade_date: { $gte: "2025-01-05", $lte: "2026-01-05" }
}).sort({ trade_date: 1 })
// 使用索引：{ symbol: 1, trade_date: 1 }
// 性能：毫秒级
```

**场景2：查询A股市场某日所有股票行情**
```javascript
db.stock_quotes.find({
  market: "A_STOCK",
  trade_date: "2026-01-05"
})
// 使用索引：{ market: 1, trade_date: 1 }
// 性能：毫秒级
```

### 2.6 数据清理策略

#### 2.6.1 分钟数据清理（每日）

```javascript
// 每天凌晨清理1周前的分钟数据
db.stock_minutes.deleteMany({
  fetched_at: { $lt: new Date(Date.now() - 7 * 24 * 60 * 60 * 1000) }
})
```

#### 2.6.2 其他数据清理

- **日线行情**：写入前清理1年前数据
- **财务报表**：写入前清理1年前数据
- **新闻资讯**：每周清理1年前数据
- **公司信息**：每周清理1年前数据

## 四、股票数据表

### 4.1 股票当日行情表（`stock_quotes`）

#### 4.1.1 表说明

存储股票的当日行情数据（轻量数据），包括开盘价、收盘价、成交量等。

**数据特点**：
- 每天更新一次（收盘后）
- 保留时长：1年
- 支持盘中数据标识（`is_complete` 字段）

#### 9.1.2 表结构

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_id` | ObjectId | ✅ | 主键 |
| `symbol` | String | ✅ | 股票代码（600000.SH） |
| `market` | String | ✅ | 市场类型：A_STOCK / US_STOCK / HK_STOCK |
| `trade_date` | String | ✅ | 交易日期（YYYY-MM-DD） |
| `open` | Decimal | ✅ | 开盘价 |
| `high` | Decimal | ✅ | 最高价 |
| `low` | Decimal | ✅ | 最低价 |
| `close` | Decimal | ✅ | 收盘价 |
| `pre_close` | Decimal | ❌ | 昨收价 |
| `volume` | Int64 | ✅ | 成交量（股） |
| `amount` | Decimal | ❌ | 成交额（元） |
| `change` | Decimal | ❌ | 涨跌额 |
| `change_pct` | Decimal | ❌ | 涨跌幅（%） |
| `turnover_rate` | Decimal | ❌ | 换手率（%） |
| `data_source` | String | ✅ | 数据来源（tushare、akshare、user_realtime） |
| `is_complete` | Boolean | ✅ | 是否完整收盘数据（true=收盘后，false=盘中） |
| `fetched_at` | Date | ✅ | 数据获取时间 |

#### 4.1.3 示例数据

```javascript
{
  _id: ObjectId("..."),
  symbol: "600000.SH",
  market: "A_STOCK",
  trade_date: "2026-01-05",

  high: 10.80,
  low: 10.30,

  volume: 1000000,
  amount: 10700000,
```

#### 4.4.4 并发访问优化策略

**场景**：多个用户同时请求同一只股票的分钟数据进行分析。

**问题**：
- 如果每个用户都调用API获取数据，会造成API配额浪费
- 如果所有用户都等待第一个用户写入数据库后再读取，会导致响应延迟

**解决方案：分布式锁 + 数据库缓存**

```
用户A、B、C同时请求 600519.SH 的分钟数据
    │
    ▼
检查数据库是否存在且未过期（TTL：1周）
    │
    ├─ 存在且未过期 → 直接返回数据库数据
    │
    └─ 不存在或已过期
        │
        ▼
    尝试获取分布式锁（Redis）
    锁键：lock:minute_data:{symbol}:{freq}
        │
        ├─ 用户A获取到锁
        │   ├─ 调用数据源API获取分钟数据
        │   ├─ 写入数据库（通道2，限流）
        │   ├─ 释放锁
        │   └─ 返回数据给用户A
        │
        └─ 用户B、C未获取到锁
            ├─ 等待2秒
            ├─ 重新检查数据库
            │   ├─ 用户A已写入 → 返回数据库数据
            │   └─ 仍未写入 → 降级到备用数据源或返回错误
            └─ 返回数据给用户B、C
```

**关键配置**：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 锁键格式 | `lock:minute_data:{symbol}:{freq}` | 按股票+频率粒度加锁 |
| 锁超时 | 30秒 | 防止死锁 |
| 等待时间 | 2秒 | 未获取锁的用户等待时间 |
| 数据TTL | 1周 | 分钟数据保留时长 |

**效果**：
- ✅ 100个用户同时请求同一股票，只调用1次API
- ✅ 后续用户直接从数据库获取，响应更快
- ✅ 避免API配额浪费
- ✅ 保证数据一致性

## 十、数据保留与清理策略

### 10.1 数据保留时长

| 数据类型 | 保留时长 | 存储范围 | 清理策略 |
|---------|---------|---------|---------|
| **股票当日行情** | **1周** | **仅自选股** | 定时任务每天清理过期数据 |
| **历史日K线** | **1周** | **仅自选股** | 写入前清理1周前的数据 |
| **分钟行情** | **1周** | **仅自选股** | 定时任务每天清理过期数据 |
| 财务报表 | 1年（保留最新4期） | 全市场 | 写入前清理1年前的数据 |
| 公司信息 | 1年 | 全市场 | 定时任务每周清理过期数据 |
| 股票基本信息 | 永久保留 | 全市场 | 不清理 |
| **新闻资讯** | **1年** | 全市场 | 定时任务每周清理过期数据 |
| **宏观经济** | **永久保留** | 全市场 | 不清理 |
| 用户自选股 | 永久保留 | 按用户 | 不清理 |
| 数据同步任务 | 30天 | 系统级 | 定时任务清理旧任务 |
| 写入限流记录 | 1分钟 | 系统级 | 自动过期 |

**重要说明**：
- **日线/分钟行情数据**：公共数据库仅存储用户自选股的数据，保留1周
- **实时日K数据需求**：用户需从自己配置的付费接口（如Tushare Pro）实时获取，系统不同步全市场日K数据
- **设计原因**：日线数据主要用于实时短线分析，非自选股无需长期存储

### 10.2 清理策略

#### 10.2.1 写入前清理（历史数据）

**适用范围**：历史日K线（自选股）、财务报表（全市场）

**日线数据清理逻辑**（自选股）：
1. 写入前先删除该股票**1周前**的数据
2. 然后插入新数据
3. 确保数据库中只保留最新1周的自选股日K数据

**财务报表清理逻辑**（全市场）：
1. 写入前先删除该股票1年前的数据
2. 然后插入新数据
3. 确保数据库中始终只保留最新1年数据

**示例**（自选股日K线）：
```
写入000001.SZ的历史K线数据（当前日期：2026-01-10）

步骤1：删除1周前的数据
DELETE FROM stock_klines
WHERE symbol = '000001.SZ'
AND trade_date < '2026-01-03'

步骤2：插入新数据
INSERT INTO stock_klines
VALUES (2026-01-03至2026-01-10的数据)
```

#### 10.2.2 定时任务清理（过期数据）

**适用范围**：股票当日行情、公司信息、历史新闻、同步任务

**清理频率**：
- 当日行情：每天凌晨1点清理
- 公司信息：每周日凌晨清理
- 历史新闻：每周日凌晨清理
- 同步任务：每天凌晨2点清理

**清理逻辑**：
```javascript
// 当日行情清理
db.stock_quotes.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})

// 公司信息清理
db.stock_companies.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})

// 历史新闻清理
db.stock_news.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})
```

#### 10.2.3 增量更新（智能对比）

**适用范围**：实时数据源写入

**更新逻辑**：
1. 从数据库读取该股票的现有数据
2. 对比接口返回的数据
3. 新增的日期 → INSERT
4. 已存在的日期 → UPDATE（如有变化）
5. 数据库有但接口没有的 → 保留（不删除历史）

**示例**：
```
场景：数据库中有000001.SZ的2024-01-01到2024-12-31的数据
     接口返回2024-01-01到2025-01-05的数据

处理流程：
1. 2024-01-01到2024-12-31：已存在，对比后如有变化则UPDATE
2. 2025-01-01到2025-01-05：新增，INSERT
3. 清理2024-01-01之前的1年旧数据
```

#### 12.1.1 必填字段验证

| 数据类型 | 必填字段 |
|---------|---------|
| 股票行情 | symbol, trade_date, open, high, low, close, volume |
| 财务报表 | symbol, report_date, report_type |
| 公司信息 | symbol, company_name, listing_date, exchange |

#### 12.1.2 数据范围验证

| 字段 | 验证规则 |
|------|---------|
| open/high/low/close | 必须 > 0 |
| volume | 必须 >= 0 |
| high | 必须 >= low |
| change_pct | -20% ~ +20%（A股单日涨跌限制） |
| trade_date | 合法日期且不早于1990-01-01 |

### 12.2 数据一致性

#### 12.2.1 价格一致性

**规则**：
- high >= open 且 high >= close
- low <= open 且 low <= close
- pre_close = 前一交易日的close

#### 12.2.2 财务报表一致性

**规则**：
- 三大报表的勾稽关系验证
- 资产 = 负债 + 所有者权益
- 净利润在利润表和资产负债表中一致

### 12.3 数据质量评分

**评分维度**：
| 维度 | 权重 | 说明 |
|------|------|------|
| 完整性 | 40分 | 必填字段完整率 |
| 合理性 | 30分 | 价格、成交量合理性 |
| 一致性 | 30分 | 价格关系一致性 |

**评分等级**：
- 90-100分：高质量
- 70-89分：中等质量
- <70分：低质量（记录警告）

## 十二、数据源状态监控表

### 12.1 数据源状态表（`data_source_status`）

#### 12.1.1 表说明

存储各市场、各数据类型的数据源健康状态，为前端监控页面提供数据支持。

#### 12.1.2 表结构

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_id` | ObjectId | ✅ | 主键 |
| `market` | String | ✅ | 市场类型：A_STOCK / US_STOCK / HK_STOCK |
| `data_type` | String | ✅ | 数据类型：daily_quote / financials / company_info 等 |
| `source_type` | String | ✅ | 数据源类型：system / user |
| `source_id` | String | ✅ | 数据源标识：tushare / akshare / yahoo 等 |
| `user_id` | ObjectId | ❌ | 用户ID（source_type=user 时必填） |

**当前状态字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `status` | String | ✅ | 状态：healthy / degraded / unavailable / standby |
| `last_check_at` | Date | ✅ | 最后检查时间 |
| `last_check_type` | String | ✅ | 检查类型：sync_task / manual_retry / tool_call |
| `response_time_ms` | Int32 | ❌ | 最后一次响应时间（毫秒） |
| `avg_response_time_ms` | Int32 | ❌ | 平均响应时间（24小时，毫秒） |

**失败信息字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `failure_count` | Int32 | ✅ | 连续失败次数 |
| `max_failure_count` | Int32 | ✅ | 最大失败次数阈值（默认3） |
| `last_error` | Object | ❌ | 最后一次错误信息 |

**last_error 对象结构**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `code` | String | 错误码 |
| `message` | String | 错误信息（用户可见） |
| `detail` | Object | 完整错误信息（管理员可见） |
| `occurred_at` | Date | 发生时间 |

**降级信息字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `is_fallback` | Boolean | ✅ | 是否为降级状态 |
| `fallback_info` | Object | ❌ | 降级详情 |

**fallback_info 对象结构**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `primary_source` | String | 主数据源ID |
| `current_source` | String | 当前数据源ID |
| `fallback_reason` | String | 降级原因 |
| `fallback_at` | Date | 降级时间 |

**接口明细字段**（一个数据类型可能包含多个接口）：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `api_endpoints` | Array | ❌ | 接口明细列表 |

**api_endpoints 数组元素结构**：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `endpoint_name` | String | 接口名称（如 pro_bar） |
| `endpoint_name_cn` | String | 接口中文名称 |
| `status` | String | 接口状态 |
| `last_check_at` | Date | 最后检查时间 |
| `failure_count` | Int32 | 失败次数 |
| `last_error` | Object | 错误信息 |

**元数据字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `created_at` | Date | ✅ | 创建时间 |
| `updated_at` | Date | ✅ | 更新时间 |

#### 12.1.3 唯一索引

```javascript
  { unique: true }

```

### 12.2 状态历史记录表（`data_source_status_history`）

#### 12.2.1 表说明

记录数据源状态的变化历史，用于状态趋势分析和问题排查。

#### 12.2.2 表结构

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_id` | ObjectId | ✅ | 主键 |
| `market` | String | ✅ | 市场类型 |
| `data_type` | String | ✅ | 数据类型 |
| `source_type` | String | ✅ | 数据源类型：system / user |
| `source_id` | String | ✅ | 数据源标识 |
| `user_id` | ObjectId | ❌ | 用户ID（source_type=user 时必填） |

**事件信息字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `event_type` | String | ✅ | 事件类型：status_changed / fallback / recovered / api_failed |
| `from_status` | String | ❌ | 变化前状态 |
| `to_status` | String | ❌ | 变化后状态 |
| `error_code` | String | ❌ | 错误码 |
| `error_message` | String | ❌ | 错误信息 |
| `response_time_ms` | Int32 | ❌ | 响应时间 |
| `check_type` | String | ❌ | 检查类型：sync_task / manual_retry / tool_call |

**接口级别事件字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `api_endpoint` | String | ❌ | 接口名称（接口级别事件时记录） |
| `from_source` | String | ❌ | 变化前数据源 |
| `to_source` | String | ❌ | 变化后数据源 |

**时间字段**：

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `timestamp` | Date | ✅ | 事件发生时间 |
