## 一、数据库概述

### 1.1 数据库选型

本项目使用 **MongoDB** 作为主要数据库，原因：
- 文档型数据库，适合存储股票行情、财务报表等结构化数据
- 支持水平扩展，便于应对海量历史数据
- 灵活的Schema设计，便于添加新字段
- 内置分片和副本集支持，保证高可用性

### 1.2 数据库命名规范

**数据库名**：`stock_analysis`

**集合命名规范**：
- 使用小写字母和下划线
- 采用 `模块_实体` 的命名方式
- 示例：`stock_quotes`、`user_data_sources`、`system_data_sources`

### 1.3 字段命名规范

**统一使用小写下划线命名法（snake_case）**：
- ✅ `trade_date`、`data_source`、`api_key`
- ❌ `tradeDate`、`dataSource`、`apiKey`

### 1.4 数据类型规范

| 数据类型 | MongoDB类型 | 说明 |
|---------|------------|------|
| 字符串 | String | 股票代码、日期、名称等 |
| 数值 | NumberDecimal | 价格、金额等高精度数值 |
| 整数 | Int32/Int64 | 成交量、数量等 |
| 日期 | Date | 时间戳 |
| 布尔 | Boolean | 开关状态 |
| 对象 | Object | 嵌套对象 |
| 数组 | Array | 列表数据 |
| 对象ID | ObjectId | 关联ID |

## 二、多市场存储策略设计

### 2.1 设计目标

本项目需要支持多市场（A股、美股、港股）的数据存储，核心设计目标：
- **数据量可控**：确保数据量在合理范围内，避免过度增长
- **查询性能**：单股票查询场景下，保证毫秒级响应
- **维护简单**：避免过度复杂的表结构，降低开发和运维成本
- **业务匹配**：符合单股票分析为主、多市场数据关联度低的业务特点

### 2.3 存储策略决策

#### 2.3.1 最终方案：统一表 + market字段

**核心决策**：采用**统一表设计**，所有市场数据存储在同一张表中，通过 `market` 字段区分市场。

**表结构示例**：
```javascript
// stock_quotes（统一存储三市场）
{
  _id: ObjectId("..."),
  symbol: "600000.SH",        // 股票代码
  market: "A_STOCK",          // 市场标识
  trade_date: "2026-01-05",   // 交易日期
  open: 10.50,
  close: 10.70,
  // ...
}

{
  _id: ObjectId("..."),
  symbol: "AAPL",             // 美股代码
  market: "US_STOCK",         // 市场标识
  trade_date: "2026-01-05",
  open: 180.20,
  close: 182.50,
  // ...
}
```

#### 2.3.2 选择统一表的理由

**理由1：数据量可控**
- 三市场合并后的日线行情数据量约437.5万条/年
- MongoDB单表可轻松处理千万级数据
- 在1年保留策略下，数据总量远未达到性能瓶颈

**理由2：业务逻辑统一**
- 同类型数据（如日线行情）的业务逻辑高度相似
- 统一表便于实现统一的数据清理、限流、技术指标计算逻辑
- 避免为每个市场重复实现相同功能

**理由3：查询性能优化**
- 主查询场景：单股票查询 `WHERE symbol = ? AND trade_date >= ? AND trade_date <= ?`
- 使用复合索引 `(symbol, trade_date)` 或 `(market, symbol, trade_date)`
- 单表查询性能不会因数据量增加而显著下降

**理由4：简化开发与运维**
- 代码中无需判断市场类型，统一处理
- 减少表数量，降低维护成本
- 便于未来扩展新市场（如加英股、日股等）

**理由5：跨市场查询需求**
- 虽然当前主要是单股票分析，但未来可能有跨市场对比需求
- 统一表便于实现"查询同一公司在A股和港股的表现"等场景

#### 2.3.3 按市场分离表的缺陷分析

**方案对比**（按市场分离表）：
```
stock_quotes_a_stock    // A股行情
stock_quotes_us_stock   // 美股行情
stock_quotes_hk_stock   // 港股行情
```

**主要缺陷**：
1. **代码复杂度高**：需要为每个市场单独实现数据获取、清理、限流逻辑
2. **维护成本高**：3倍的表数量，3倍的索引，3倍的清理任务
3. **跨市场查询困难**：需要UNION查询多个表，性能差
4. **过度设计**：数据量远未达到需要分表的程度

### 2.4 差异化保留策略

基于数据量特点，采用差异化保留策略：

| 数据类型 | 存储表 | 保留时长 | 策略说明 |
|---------|-------|---------|---------|
| 日线行情 | `stock_quotes` | 1年 | 统一表，全市场合并存储 |
| 日K线+指标 | `stock_klines` | 1年 | 统一表，包含技术指标 |
| **分钟行情** | `stock_minutes` | **1周** | 统一表，仅短期缓存，避免数据量爆炸 |
| 财务报表 | `stock_financials` | 1年（4期） | 统一表，仅保留最新4期 |
| 公司信息 | `stock_companies` | 1年 | 统一表，定期更新 |
| 新闻资讯 | `stock_news` | 1年 | 统一表，定期清理 |
| 宏观经济 | `macro_economic` | 永久 | 不分市场，数据量极小 |

**核心原则**：
- 高频大容量数据（分钟行情）短期缓存
- 低频小容量数据（财务、公司、新闻）长期保留
- 通过 `(market, symbol, trade_date)` 复合索引保证查询性能

### 2.5 索引优化设计

#### 2.5.1 核心索引

```javascript
// 单股票历史查询（最常见）
db.stock_quotes.createIndex({ symbol: 1, trade_date: -1 })

// 单市场查询
db.stock_quotes.createIndex({ market: 1, trade_date: -1 })

// 单股票单市场查询
db.stock_quotes.createIndex({ market: 1, symbol: 1, trade_date: -1 })
```

#### 2.5.2 查询性能保证

**场景1：查询单股票1年K线数据**
```javascript
db.stock_klines.find({
  symbol: "600000.SH",
  trade_date: { $gte: "2025-01-05", $lte: "2026-01-05" }
}).sort({ trade_date: 1 })
// 使用索引：{ symbol: 1, trade_date: 1 }
// 性能：毫秒级
```

**场景2：查询A股市场某日所有股票行情**
```javascript
db.stock_quotes.find({
  market: "A_STOCK",
  trade_date: "2026-01-05"
})
// 使用索引：{ market: 1, trade_date: 1 }
// 性能：毫秒级
```

### 2.6 数据清理策略

#### 2.6.1 分钟数据清理（每日）

```javascript
// 每天凌晨清理1周前的分钟数据
db.stock_minutes.deleteMany({
  fetched_at: { $lt: new Date(Date.now() - 7 * 24 * 60 * 60 * 1000) }
})
```

#### 2.6.2 其他数据清理

- **日线行情**：写入前清理1年前数据
- **财务报表**：写入前清理1年前数据
- **新闻资讯**：每周清理1年前数据
- **公司信息**：每周清理1年前数据

## 三、数据源配置表

### 3.1 系统公共数据源表（`system_data_sources`）

#### 4.1.1 表说明

存储超级管理员配置的系统级公共数据源，供所有用户共享使用。

#### 9.1.2 表结构

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_id` | ObjectId | ✅ | 主键 |
| `source_id` | String | ✅ | 数据源标识（tushare、akshare等） |

| `enabled` | Boolean | ✅ | 是否启用 |
| `priority` | Int32 | ✅ | 优先级（数字越小优先级越高） |
| `is_system_public` | Boolean | ✅ | 标识为系统公共配置（固定为true） |
| `config` | Object | ✅ | 数据源配置（加密存储） |
| `rate_limit` | Object | ❌ | 限流配置 |
| `supported_data_types` | Array | ✅ | 支持的数据类型 |
| `metadata` | Object | ✅ | 元数据 |

**config 对象结构**：
| 字段名 | 类型 | 说明 |
|--------|------|------|
| `api_key` | String | API密钥（加密存储） |

|--------|------|------|

| `max_requests_per_hour` | Int32 | 每小时最大请求数 |

**metadata 对象结构**：

|--------|------|------|
| `created_at` | Date | 创建时间 |
| `updated_at` | Date | 更新时间 |
| `last_tested_at` | Date | 最后测试时间 |
| `test_status` | String | 测试状态：success / failed |
| `last_error` | String | 最后错误信息 |

#### 2.1.3 示例数据

```javascript
{
  _id: ObjectId("..."),

  market: "A_STOCK",

  priority: 1,
  is_system_public: true,

  config: {
    api_key: "encrypted_system_token_here",
    base_url: "https://api.tushare.pro",

  },

  rate_limit: {
    max_requests_per_minute: 1000,
    max_requests_per_hour: 10000,
    max_requests_per_day: 100000
  },

  supported_data_types: ["quote", "history", "financials", "company"],

  metadata: {
    created_at: ISODate("2026-01-01T00:00:00Z"),
    updated_at: ISODate("2026-01-01T00:00:00Z"),
    last_tested_at: ISODate("2026-01-05T10:00:00Z"),

    last_error: null
```

## 四、股票数据表

### 4.1 股票当日行情表（`stock_quotes`）

#### 4.1.1 表说明

存储股票的当日行情数据（轻量数据），包括开盘价、收盘价、成交量等。

**数据特点**：
- 每天更新一次（收盘后）
- 保留时长：1年
- 支持盘中数据标识（`is_complete` 字段）

#### 9.1.2 表结构

| 字段名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `_id` | ObjectId | ✅ | 主键 |
| `symbol` | String | ✅ | 股票代码（600000.SH） |
| `market` | String | ✅ | 市场类型：A_STOCK / US_STOCK / HK_STOCK |
| `trade_date` | String | ✅ | 交易日期（YYYY-MM-DD） |
| `open` | Decimal | ✅ | 开盘价 |
| `high` | Decimal | ✅ | 最高价 |
| `low` | Decimal | ✅ | 最低价 |
| `close` | Decimal | ✅ | 收盘价 |
| `pre_close` | Decimal | ❌ | 昨收价 |
| `volume` | Int64 | ✅ | 成交量（股） |
| `amount` | Decimal | ❌ | 成交额（元） |
| `change` | Decimal | ❌ | 涨跌额 |
| `change_pct` | Decimal | ❌ | 涨跌幅（%） |
| `turnover_rate` | Decimal | ❌ | 换手率（%） |
| `data_source` | String | ✅ | 数据来源（tushare、akshare、user_realtime） |
| `is_complete` | Boolean | ✅ | 是否完整收盘数据（true=收盘后，false=盘中） |
| `fetched_at` | Date | ✅ | 数据获取时间 |

#### 4.1.3 示例数据

```javascript
{
  _id: ObjectId("..."),
  symbol: "600000.SH",
  market: "A_STOCK",
  trade_date: "2026-01-05",
  open: 10.50,
  high: 10.80,
  low: 10.30,
  close: 10.70,
  pre_close: 10.40,
  volume: 1000000,
  amount: 10700000,
  change: 0.30,
  change_pct: 2.88,
  turnover_rate: 0.5,
  data_source: "tushare",
  is_complete: true,  // 收盘后完整数据
  fetched_at: ISODate("2026-01-05T15:30:00Z")
}
```

## 十、数据保留与清理策略

### 10.1 数据保留时长

| 数据类型 | 保留时长 | 清理策略 |
|---------|---------|---------|
| 股票当日行情 | 1年 | 定时任务每天清理过期数据 |
| 历史日K线 | 1年 | 写入前清理1年前的数据 |
| **分钟行情** | **1周** | 定时任务每天清理过期数据 |
| 财务报表 | 1年（保留最新4期） | 写入前清理1年前的数据 |
| 公司信息 | 1年 | 定时任务每周清理过期数据 |
| 股票基本信息 | 永久保留 | 不清理 |
| **新闻资讯** | **1年** | 定时任务每周清理过期数据 |
| **宏观经济** | **永久保留** | 不清理 |
| 用户自选股 | 永久保留 | 不清理 |
| 数据同步任务 | 30天 | 定时任务清理旧任务 |
| 写入限流记录 | 1分钟 | 自动过期 |

### 10.2 清理策略

#### 10.2.1 写入前清理（历史数据）

**适用范围**：历史日K线、财务报表

**清理逻辑**：
1. 写入前先删除该股票1年前的数据
2. 然后插入新数据
3. 确保数据库中始终只保留最新1年数据

**示例**：
```
写入000001.SZ的历史K线数据（当前日期：2026-01-05）

步骤1：删除1年前的数据
DELETE FROM stock_klines
WHERE symbol = '000001.SZ'
AND trade_date < '2025-01-05'

步骤2：插入新数据
INSERT INTO stock_klines
VALUES (2025-01-05至2026-01-05的数据)
```

#### 10.2.2 定时任务清理（过期数据）

**适用范围**：股票当日行情、公司信息、历史新闻、同步任务

**清理频率**：
- 当日行情：每天凌晨1点清理
- 公司信息：每周日凌晨清理
- 历史新闻：每周日凌晨清理
- 同步任务：每天凌晨2点清理

**清理逻辑**：
```javascript
// 当日行情清理
db.stock_quotes.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})

// 公司信息清理
db.stock_companies.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})

// 历史新闻清理
db.stock_news.deleteMany({
  fetched_at: {$lt: new Date(Date.now() - 365 * 24 * 60 * 60 * 1000)}
})
```

#### 10.2.3 增量更新（智能对比）

**适用范围**：实时数据源写入

**更新逻辑**：
1. 从数据库读取该股票的现有数据
2. 对比接口返回的数据
3. 新增的日期 → INSERT
4. 已存在的日期 → UPDATE（如有变化）
5. 数据库有但接口没有的 → 保留（不删除历史）

**示例**：
```
场景：数据库中有000001.SZ的2024-01-01到2024-12-31的数据
     接口返回2024-01-01到2025-01-05的数据

处理流程：
1. 2024-01-01到2024-12-31：已存在，对比后如有变化则UPDATE
2. 2025-01-01到2025-01-05：新增，INSERT
3. 清理2024-01-01之前的1年旧数据
```

## 十二、数据完整性保障

### 12.1 数据验证

#### 12.1.1 必填字段验证

| 数据类型 | 必填字段 |
|---------|---------|
| 股票行情 | symbol, trade_date, open, high, low, close, volume |
| 财务报表 | symbol, report_date, report_type |
| 公司信息 | symbol, company_name, listing_date, exchange |

#### 12.1.2 数据范围验证

| 字段 | 验证规则 |
|------|---------|
| open/high/low/close | 必须 > 0 |
| volume | 必须 >= 0 |
| high | 必须 >= low |
| change_pct | -20% ~ +20%（A股单日涨跌限制） |
| trade_date | 合法日期且不早于1990-01-01 |

### 12.2 数据一致性

#### 12.2.1 价格一致性

**规则**：
- high >= open 且 high >= close
- low <= open 且 low <= close
- pre_close = 前一交易日的close

#### 12.2.2 财务报表一致性

**规则**：
- 三大报表的勾稽关系验证
- 资产 = 负债 + 所有者权益
- 净利润在利润表和资产负债表中一致

### 12.3 数据质量评分

**评分维度**：
| 维度 | 权重 | 说明 |
|------|------|------|
| 完整性 | 40分 | 必填字段完整率 |
| 合理性 | 30分 | 价格、成交量合理性 |
| 一致性 | 30分 | 价格关系一致性 |

**评分等级**：
- 90-100分：高质量
- 70-89分：中等质量
- <70分：低质量（记录警告）
