# 数据源融合与数据库设计方案

## 1. 设计理念

为了实现 AKShare (AK) 和 TuShare (TU) 数据源的统一管理，我们将采用 **"统一模型，分层存储"** 的设计思路。

*   **统一模型 (Unified Model)**: 无论数据来自 TU 还是 AK，在数据库中都存储为统一的字段格式。
*   **按业务分类 (Domain Driven)**: 不按数据源分表，而是按数据类型（如：基本面、行情、财务、新闻）分表。
*   **主键策略**: 统一使用 `symbol` (标准化股票代码，如 `000001.SH`) 作为股票唯一标识，格式为 `{原始代码}.{交易所}`。`trade_date` (如 `20240101`) 作为时间标识。

## 3. 故障切换策略 (Failover Strategy)

为了保证数据获取的高可用性，我们设计了基于**数据类型粒度**的故障切换机制。

### 3.1 切换逻辑
1.  **粒度控制**: 切换是以 **数据类型 (data_type)** 为单位的，而不是整个数据源。
    *   *场景示例*: 如果 TU 的分钟数据接口 (`stk_mins`) 挂了，系统只将 `minute_data` 类型切换到 AK (`stock_zh_a_hist_min_em`)。此时，日线数据 (`daily_data`) 仍然继续使用 TU。
2.  **优先级**: 默认优先使用 **TU (Tushare)**，因为其数据规范性更好。
3.  **自动降级**: 当检测到 TU 接口连续失败（如超时、返回空或报错）达到阈值（如 3 次）时，将 `is_using_backup` 标记为 `true`，系统自动切换调用 AK 接口。
4.  **自动恢复**: 系统会定期（如每小时）尝试探测主源 (TU) 的可用性，一旦恢复，自动切回主源。

### 3.2 监控流程
*   每次调用接口前，检查 `api_monitor` 表中对应 `data_type` 的 `is_using_backup` 状态。
*   如果 `is_using_backup = true`，直接调用 AK 接口。
*   如果 `is_using_backup = false`，调用 TU 接口。如果调用失败，记录 `fail_count`，超过阈值则更新状态并切换。

### 3.3 公共数据源与用户数据源的降级策略

**公共数据源降级**（管理员定时同步任务）：
- 公共 TU 失败时，降级到公共 AK
- 定期探测公共 TU 恢复，成功后切换回公共 TU

**用户数据源降级**（用户实时请求）：
- 用户 TU 失败时，降级到公共数据库读取
- 用户无法调用公共 TU 接口，只能从数据库读取或使用自己的 TU
- 定期探测用户 TU 恢复，成功后切换回用户 TU

**数据更新机制**：
- 用户实时获取的数据会更新公共数据库（增量更新）
- 管理员定时同步任务更新公共数据库
- 所有数据统一存储在公共数据库

## 4. 实现建议





1.  **数据清洗层**: 编写 Python 脚本 (`DataCleaner`)，专门负责将 AK 和 TU 的原始 JSON 数据转换为上述标准格式。
2.  **单位对齐**: AKShare 和 TuShare 的金额单位可能不同（如元 vs 千元 vs 万元），入库前必须统一（建议统一为**元**）。
3.  **日期格式**: 统一使用 `YYYYMMDD` 字符串作为日期字段，方便查询和索引。
4.  **数据优先级**:
    *   如果字段在两个源都有（如 `open`, `close`），建议优先使用 **TuShare** (数据更规范)，**AKShare** 作为补充或校验。
